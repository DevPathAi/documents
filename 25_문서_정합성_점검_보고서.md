## 1. 점검 개요

본 보고서는 두 가지 질문에 답한다.

1. **문서 간 정합성** — `documents` 레포 내부 문서들과 프로젝트 설계 산출물이 서로 모순 없이 일치하는가.
2. **문서 ↔ 레포 정합성** — 그 문서가 정의한 설계대로 실제 코드/인프라 레포가 구축되었는가.

점검 시점은 **W1(2026-06-15~06-19) 착수 직전**이다. 현재 백엔드 5개 서비스는 모두 **인프라 부트스트랩 스켈레톤 상태**(`Application` 클래스 + `DbConnectionTest` + 설정 파일만 존재, 도메인 코드 미착수)이므로, 본 점검은 "구현 완성도"가 아니라 **설계 의도와 골격·문서의 일치 여부**에 초점을 둔다.

---

## 2. 종합 판정

**정합성 관리 체계는 성숙하다.** W1 재정의 설계서가 폴리레포·DB 전환 드리프트를 명시적으로 인지하고, Synapse #9·#12 가드("코드-문서 동일 PR")까지 적용하고 있다. 다만 **현행 W1 문서 동기화 스코프가 커버하지 못하는 잔존 불일치 7건**이 확인된다. 그중 1건(아키텍처 정의서의 모놀리식 서술)은 W1 종료 후에도 해소되지 않는 사각지대다.

| 구분 | 항목 수 | 상태 |
|---|:---:|---|
| ✅ 정합 확인 (검증 완료) | 5 | 문서-레포 일치, 조치 불필요 |
| 🔴 Critical | 1 | W1 미커버 · 근본 서술 모순 |
| 🟠 Major | 2 | W1 스코프 확장 또는 즉시 갱신 필요 |

## 4. 잔존 불일치 (심각도순)

### 🔴 D1. 아키텍처 정의서(03)가 여전히 "모놀리식 코어"를 전제 — *W1 미커버*

**현상**: `03_프로젝트_아키텍처_정의서`는 시스템 스타일을 모놀리식으로 정의하나, 실제는 **폴리레포 마이크로서비스 9개**다.

| 위치 | 문서 서술 | 실제 |
|---|---|---|
| 03 §머리말 | 스타일 = "모놀리식 코어 + 격리된 Sandbox 서비스 + AI Gateway" | 폴리레포(독립 배포 서비스 5개 + gateway + shared) |
| 03 §1.2-1 | "팀 규모(12명)에서 MSA는 과도. Sandbox만 격리" | 1인 창업자 / 전 서비스 분리 배포 |
| 03 §2 | `devpath-api/` 단일 앱 모듈 트리(user·learning·content·community…) | `platform/learning/community-svc` 등 별도 레포로 분할 |
| 03 ADR-009 | "모놀리식 코어 — 팀 규모 대비 MSA 과도" | 결정이 번복됨(폴리레포) |

**왜 사각지대인가**: [W1 재정의 설계서](https://github.com/DevPathAi/devpath-shared/blob/main/docs/superpowers/specs/2026-06-13-w1-infra-redefinition-design.md) §2는 "모노리포 → 폴리레포(재정의)"를 **배경으로만 인지**하고, 문서 동기화 작업(§⑤ / plan Task 7)은 **MySQL→PostgreSQL 표기 갱신에 한정**된다. 즉 03의 모놀리식 서술·`devpath-api` 트리·ADR-009는 **W1 종료 후에도 그대로 남는다.**

**부수 모순**: 03 §1.2가 전제하는 "팀 12명"은 `MVP_스펙`의 "1인 창업자 단독 개발" 및 피치 내러티브("1인 풀스택")와 충돌한다. 03은 내부 개발 문서이나, 심사·멘토 공유 시 토폴로지·인력 전제가 피치와 어긋나면 신뢰 리스크가 된다.

**조치**: §6-2 참조 (03 재서술).

### 🟠 D2. org 프로필 README가 서비스 레포를 "예정"으로 기술

**현상**: `.github/profile/README.md`의 레포 표는 `documents`·`storyboard`·`prototype`·`templates` 4개만 노출하고, 서비스 레포 분리를 **미래형**으로 서술한다.

> (인용) "본격 개발 단계에서는 도메인별 서비스 레포로 분리해 운영할 **예정입니다**" → `devpath-*-svc`, `devpath-gateway`, `devpath-shared`, `devpath-frontend`, `devpath-gitops`

이 레포들은 **이미 전부 생성·푸시 완료**(2026-06-13) 상태다. 공개 조직 랜딩 페이지가 현실보다 뒤처진 stale 상태.

**조치**: §6-3.

---

### 🟠 D3. MySQL→PostgreSQL 문서 드리프트가 W1 검증 스코프보다 넓음

**현상**: W1 성공 기준 5는 "**02·03**에 MySQL 잔존 0건"만 검증한다. 그러나 `02_ERD`·`03_아키텍처`는 이미 PostgreSQL 표기다(03 ADR-002 = "PostgreSQL(SSOT) + pgvector 분리"). **실제 MySQL 잔존은 검증 범위 밖 문서들에 집중**되어 있다.

| 문서 | MySQL 잔존 내용(요약) |
|---|---|
| `01_프로젝트_계획서` | DB 항목 "MySQL 8.x (Source of Truth)" |
| `05_화면_흐름_시퀀스_다이어그램` | mermaid participant "DB as MySQL" (2개소) |
| `10_환경_설정_템플릿` | compose 전체가 `mysql:8.0`, `jdbc:mysql://`, `MYSQL_*` 환경변수 (다수) |
| `11_테스트_전략서` | Testcontainers "MySQL", `mysql:8` 이미지 |
| `16_운영_메뉴얼` | "MySQL/pgvector/Redis/ES", "MySQL(SSOT) RDS 스냅샷" |
| `Home.md` | Data 스택 "MySQL (SSOT)" |

현행 성공 기준대로면 이들은 잡히지 않는다. (plan Task 7 Step 1의 `grep -rn -i mysql *.md`는 전체를 훑지만, Step 2 갱신 대상과 성공 기준은 02·03으로 한정 — 실행 시 누락 위험)

**조치**: §6-1 (스코프 확장).

---

### 🟡 D4. 프로젝트 산출물 vs 레포 문서의 정본(canonical) 충돌 — AR

**현상**: 프로젝트 작업본 `devpath-ai-architecture-v2.html`·`devpath-ai-ar-expansion.md`는 **AR을 현재형 핵심 Pillar**로 제시한다(네임스페이스 `core/ar/community` 분리, Scene/Asset/Realtime 서비스 도해). 반면 레포 `documents`는 AR을 아카이브(`22_…참고설계`)로 강등했다(= C3과 동일 사실의 이면).

내부적으로 레포는 일관되나, **두 정본이 공존**한다. 피치에서 arch-v2를 쓰면서 레포는 "AR 보류"라고 하면 심사자에게 모순으로 보인다.

**조치**: §6-4 (정본 단일화).
