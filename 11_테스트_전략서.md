## 2. 레이어별 전략

### 2.1 도메인 단위 테스트
- **대상**: Aggregate, Value Object, Domain Service
- **도구**: JUnit 5 + AssertJ + **jqwik**
- **우선 영역**:
  - 적응형 진단 알고리즘 (난이도 +0.1 / -0.05 수렴)
  - 학습 경로 rationale 빌더
  - 스트릭 계산 (자정 경계, TZ)
  - 피어 매칭 스코어링
  - 멘토 컨텍스트 스냅샷 조립
  - **커뮤니티 도메인** (Eng 리뷰 보강):
    - 평판 계산 (일일 40점 상한, 7일 신규 계정 제한)
    - 에스컬레이션 상태 전이 (NONE → ESCALATED → BOUNTY_PLACED → RESOLVED)
    - 현상금 차감/환불 로직 (race condition 방지: optimistic locking 필수)
    - AI 시드 생성 파이프라인 (타임아웃, 프롬프트 인젝션 방어, refusal 처리)
    - 모더레이션 심각도 판정 + 규칙 기반 fallback
    - 학습 맥락 스냅샷 수집 (opt-in 검증, 민감 정보 필터링)

```java
@Property
void adaptiveDifficulty_correctAnswer_raisesByPoint1(
    @ForAll @DoubleRange(min = 0.0, max = 0.9) double current) {
  AdaptiveState s = AdaptiveState.of(current);
  AdaptiveState next = s.onCorrect();
  assertThat(next.difficulty()).isEqualTo(Math.min(1.0, current + 0.1));
}
```

### 2.2 통합 테스트
- **도구**: Spring Boot Test + Testcontainers (PostgreSQL, Redis, Kafka, pgvector, ES) + WireMock
- **필수 시나리오**:

| 영역 | 시나리오 |
|------|----------|
| OAuth2 | GitHub 콜백 → User INSERT + Identity + UserRegisteredEvent |
| GitHub Worker | Rate limit 처리 + 401 재시도 |
| Path Engine | Claude 응답 → pgvector 매칭 → DB + Outbox |
| Sandbox | 제출 → 풀 할당 → 실행 → 결과 저장 + AI Review 이벤트 |
| Mentor | 세션 생성 시 context_snapshot 정확 주입 |
| Outbox | dedup_key 중복 시도 → 1회만 반영 |

### 2.3 E2E 테스트
- **Web**: Playwright (TypeScript)
- **Mobile**: Flutter `integration_test` + 실기기 회귀 (Android·iOS 각 3종)
- **범위**: Smoke (배포 전 5분) / UAT (릴리즈 전 30분)

**Smoke Golden Path**
1. 랜딩 → 비회원 진단 15문항 → 로그인 게이트 → GitHub OAuth → 경로 수신 (1st Aha)
2. 콘텐츠 열기 → Sandbox 제출 → AI 리뷰 수신 (2nd Aha)
3. AI 멘토 질문 → 컨텍스트 참조 확인
4. 커뮤니티 Q&A 작성 → 학습 맥락 자동 첨부 확인

## 5. Sandbox 보안 테스트 (필수)

### 5.1 격리 탈출 시도

```java
@Test
@DisplayName("/proc 접근 시도 → 차단")
void procAccess_blocked() {
  var code = "cat /proc/1/status";
  var result = sandbox.run(code, "bash");
  assertThat(result.exitCode()).isNotZero();
  assertThat(result.stderr()).contains("Permission denied");
}

@Test
@DisplayName("아웃바운드 네트워크 차단")
void networkBlocked() {
  var code = """
    Socket s = new Socket("example.com", 80);
    """;
  var result = sandbox.run(code, "java21");
  assertThat(result.stderr()).contains("No route to host");
}

@Test
@DisplayName("30초 초과 시 SIGKILL")
void timeout_killed() {
  var code = "while(true){}";
  var result = sandbox.run(code, "java21");
  assertThat(result.status()).isEqualTo(Status.KILLED);
  assertThat(result.durationMs()).isBetween(30_000L, 32_000L);
}

@Test
@DisplayName("CPU 100% 반복 패턴 → 사용자 일시 정지 플래그")
void cpuAbuse_flagsUser() {
  for (int i = 0; i < 5; i++) {
    sandbox.run("while(true){}", "java21");
  }
  assertThat(userService.isSandboxSuspended(userId)).isTrue();
}
```

## 16. 골든 테스트 케이스 구축 방법론

AI 코드 리뷰 품질 보장을 위한 골든 케이스 50개 기준.

### 16.1 구성
| 카테고리 | 건수 | 예시 |
|---------|------|------|
| JPA/Hibernate | 10 | N+1, LazyInit, fetch join 페이징 |
| Spring Security | 5 | CSRF 누락, 권한 에스컬레이션 |
| 동시성 | 5 | Race condition, 데드락, @Transactional 전파 |
| API 설계 | 10 | RESTful 위반, 페이지네이션, 에러 응답 |
| 코드 스타일 | 10 | 네이밍, 불필요한 복잡도, 매직 넘버 |
| 보안 | 5 | SQL 인젝션, XSS, 민감정보 로깅 |
| 성능 | 5 | 불필요한 쿼리, 캐시 미사용, 대량 데이터 처리 |

### 16.2 관리
- 골든 케이스는 `test/golden/` 디렉토리에 입력 코드 + 기대 리뷰 포인트 쌍으로 저장
- 프롬프트 버전 변경 시 전수 회귀 테스트 실행
- 합격 기준: 50개 중 45개(90%) 이상 핵심 포인트 검출
- 분기 1회 골든 케이스 갱신 (신규 패턴 추가, 퇴역 케이스 제거)
