# 18. 온보딩 마이크로카피 가이드

> **목적**: 가입부터 학습 습관화까지 각 이탈 지점에서 사용자의 심리적 저항을 줄이는 마이크로카피 모음
> **적용 원칙**:
> - 개발자 타겟 → 감성 과잉 금지, 이모티콘 남용 금지, 존댓말 유지하되 간결하게
> - "마케팅 문구"가 아니라 "사용자가 다음 행동을 하게 만드는 문장"
> - 모든 카피는 **A안(권장) / B안(대안)** 으로 제시 → A/B 테스트 가능

## 1. 마이크로카피 작성 원칙

### 1.1 개발자 타겟 특화 원칙

| 하지 말 것 | 대신 할 것 |
|-----------|-----------|
| "축하합니다! 🎉🎊 환영해요!!" | "가입 완료. 2분 안에 첫 학습 경로를 받습니다." |
| "어떤 꿈을 이루고 싶으신가요?" | "이번에 달성하려는 목표는?" |
| "당신은 최고의 개발자가 될 거예요!" | "당신의 GitHub 기록을 분석해 경로를 맞춤화합니다." |
| "로그인하세요" (동사 명령) | "GitHub으로 계속" (결과 중심) |
| "조금만 기다려주세요 :)" | "학습 경로 생성 중 · 5초" |

### 1.2 5가지 황금 규칙

1. **숫자로 기대치 설정** — "잠시만요" → "15초" / "3문항" / "2분"
2. **"왜"를 한 문장 안에 넣기** — "권한을 허용해 주세요" → "GitHub 공개 저장소를 읽어 맞춤 경로를 만듭니다"
3. **다음 행동을 주어로** — "확인" → "학습 시작" / "경로 받기" / "건너뛰기"
4. **부정문 대신 긍정문** — "실패했습니다" → "다시 시도하면 됩니다"
5. **사용자 언어 사용** — 개발자는 "과정", "코스"보다 "로드맵", "스택", "프로젝트", "레포"에 반응

### 4.1 온보딩 진입 화면 (기대치 설정)

**A안 (권장 · 투명성)**
> **4개의 질문, 2분 소요**
> 답변은 나중에 마이페이지에서 언제든 수정 가능합니다.

>
> [시작하기 →]   [나중에]

**B안 (짧게)**
> 2분이면 끝납니다.
> 질문은 총 4단계, 중간 이탈해도 이어서 진행할 수 있습니다.

### 4.2 진행률 표시 (모든 화면 상단 고정)

**A안 (권장)**
```
●●○○  2/4 · 1분 30초 남음
```

**B안 (숫자만)**
```
2 / 4
```

**피해야 할 표시**
- `50%` (퍼센트만 있으면 남은 양이 체감 안 됨)
- 진행률 막대만 있고 숫자 없음

### 4.3 질문 1: 학습 목표

> **어떤 목표로 학습하시나요?**
> 개인화 경로의 방향을 결정합니다.
>
> - 취업 준비 — 첫 직장, 부트캠프 수료 직후
> - 이직·레벨업 — 현직자, 더 큰 도전
> - 실무 역량 강화 — 현재 업무에 바로 적용
> - 사이드 프로젝트 — 개인 포트폴리오, 창업 준비
>
> 선택은 하나만. 부가 관심사는 다음 단계에서.

### 4.4 질문 2: 관심 트랙

> **어떤 트랙을 중점적으로?**
> 중복 선택 가능 (최대 2개)
>
> - 백엔드 (Spring · JPA · MSA)
> - 프론트엔드 (React · Next.js)
> - 모바일 (Flutter · iOS · Android)
> - DevOps·인프라 (Docker · K8s)
> - 풀스택 — 백엔드 + 프론트 동시
>
> 나중에 다른 트랙도 추가할 수 있습니다.

> 💡 **"풀스택"을 아래로 두는 이유**: 초보는 "풀스택"을 쉽게 누르지만 실제로는 하나에 집중하는 게 성장에 좋습니다. 트랙을 좁혀야 콘텐츠 추천 품질도 올라갑니다.

### 4.5 질문 3: 진단 테스트 진입 (도입 화면)

> **실력 진단: 15문항**
>
> - 모르면 "잘 모르겠어요"를 눌러도 됩니다. (감점 없음)
> - 쉬운 문제로 시작해 정답 시 난이도가 올라갑니다.
> - 중간 이탈해도 진행 상황은 저장됩니다.
>
> [진단 시작 →]

**"잘 모르겠어요" 옵션의 중요성**
> 개발자는 찍기를 싫어합니다. 모르는 걸 모른다고 하는 게 정확한 진단이고, 사용자에게도 심리적 안전을 줍니다.

### 4.6 진단 문항 화면

**문항 예시**
> **Q3. 다음 코드에서 발생할 수 있는 문제는?**
> ```java
> List<Order> orders = orderRepository.findAll();
> orders.forEach(o -> System.out.println(o.getMember().getName()));
> ```
>
> - N+1 쿼리 발생
> - LazyInitializationException 가능
> - 둘 다 가능성 있음
> - 문제 없음
> - **잘 모르겠어요** ← 항상 맨 아래 고정

**문항 간 전환 문구 (옵션 · 격려형)**
- 5문항 완료 시: "1/3 지점. 지금까지 답변이 맞춤 경로 품질을 높이고 있습니다."
- 10문항 완료 시: "절반을 넘었습니다. 마지막까지 솔직하게."

> 💡 격려 문구는 **3문항마다 한 번**이면 충분합니다. 매 문항마다 나오면 피로합니다.

### 4.7 질문 4: 학습 약속

> **주당 몇 시간 학습할 수 있나요?**
> 솔직하게. 과대 설정은 이탈 원인 1위입니다.

> - 4-7시간 — 평일 저녁 중심
> - 8-14시간 — 적극적으로
> - 15시간 이상 — 부트캠프 수준

**보조 질문 (같은 화면)**
> 선호 학습 시간대는? (알림을 이때 보냅니다)
> [ 아침 ]  [ 낮 ]  [ 저녁 ]  [ 밤 ]

**A안 (권장 · 투명한 진행)**

> 거의 다 왔어요. 5개 중 2개 통과.

**절대 쓰지 말 것**
- "실패!" (부정적 감정 유발)
- "틀렸습니다" (정오답 이분법)
- "다시 시도하세요" (다음 행동 불명확)

### 7.4 AI 코드 리뷰 결과 (톤 · 3단 구조)

> **코드 리뷰 · Claude**
>
> **잘한 점**
> `@EntityGraph` 사용으로 N+1 해결. 깔끔합니다.
>
> **개선 제안**
> - `findAll()` 대신 페이징을 고려해보세요. 100건이 넘어가면 메모리 이슈 가능.
> - 테스트에 `@Transactional(readOnly = true)`가 누락되어 있습니다.
>
> **실무 관점**
> 실제 프로덕션에서는 이런 리포지토리에 캐시 전략(Redis)을 더하는 경우가 많습니다.

> 💡 **"잘한 점 → 개선 → 실무 관점" 3단 구조**가 좌절감을 줄이고 성장감을 줍니다.

## 부록 A. 톤 앤 보이스 치트시트

### 인칭
- 1인칭: "우리" 지양 → "DevPath AI가" 또는 "AI 멘토가"
- 2인칭: "당신" (한국어에서 부자연스러울 땐 생략) 또는 이름

### 시제
- 미래 약속: "~됩니다" (X "~될 것입니다")
- 과거 완료: "~했어요" (친근) / "~완료되었습니다" (공식)

### 동사

| 피해야 할 동사 | 대안 |
|---------------|------|
| "해주세요" (명령) | "할 수 있어요" (제안) |
| "필요합니다" | "~이면 도움 됩니다" |
| "해야 합니다" | "추천드립니다" / 그냥 정보 제시 |

### 이모지 사용 기준
- **허용**: 기능 라벨 (📊 리포트 · 🔥 스트릭 · ✨ 코드 리뷰)
- **지양**: 감정 표현 과잉 (🎉🎊🥳✨)
- **금지**: 문장 끝 장식 이모지 (`~했어요 😊`)

### 길이
- 버튼: 2~4어절
- 헤드라인: 한 줄
- 설명: 두 문장 이내. 필요하면 리스트로.

## 부록 B. 측정 가이드 (A/B 테스트 우선순위)
