## 1. 배경과 목적

37·38번(2026-06-19) 이후 약 2주간 슬라이스 7~9(AI 멘토·커뮤니티·LCS), 알림 모듈의 notification-svc 이관, 평판 엔진 Build 1~3, 남용방지 Build 1~3 등 대규모 구현이 진행되어 문서·코드 정합성이 다시 벌어졌을 가능성이 높다(메모리 기록: "문서가 실제 레포보다 뒤처짐").

이 작업은 워크스페이스 전체를 대상으로 **2차 전수 정합성 점검**을 수행하고, 결과를 **우선순위별 리팩토링(문서 수정) 실행계획**으로 재구성한다.

## 2. 결정 사항 (사용자 확인 완료)

| 결정 항목 | 선택 |
|---|---|
| 리팩토링 대상 | 문서 + 코드 정합성 (37·38의 후속 성격) |
| 기준 브랜치 | 각 레포 `origin/develop` (feature 브랜치는 "진행 중"으로만 기록) |
| 진실 기준 | **코드가 진실** — 문서를 코드에 맞춤, 목표 설계는 TARGET 라벨 분리, 코드 수정은 계획에 불포함 |
| 점검 범위 | 워크스페이스 전체 (37번과 동일: documents + 서비스 레포 13개 + storyboard/prototype/workflow-guide/workflow-dashboard/templates) |
| 산출물 형태 | 점검 보고서 + 실행계획서 2문서 (37·38 방식 계승) |

## 4. 점검 절차 (5단계)

### Phase 0 — 기준 확정
- 전 레포 `git fetch origin` 후 `origin/develop`을 점검 기준으로 고정
- 현재 체크아웃된 feature 브랜치는 건드리지 않음 — `git show origin/develop:<path>` 또는 `git ls-tree origin/develop` 방식으로 읽음
- `origin/develop`이 없는 레포는 기본 브랜치를 기준으로 쓰고 42번에 명기
- 각 레포의 미머지 feature 브랜치를 "진행 중 작업" 표로 수집

### Phase 1 — 회귀 체크 (C)
- 38번의 P0-1~P0-5, P1-1~P1-11, P2-1~P2-4, I-1~I-3 각각이 실제 반영됐는지 항목별 재검증

### Phase 2 — 사실 수집 (A) — 읽기 전용 에이전트 5개 병렬
| 그룹 | 대상 | 산출 |
|---|---|---|
| ① 백엔드 svc | platform/learning/ai/community/sandbox/lcs/notification (7개) | 엔드포인트 표, 주요 도메인 모듈, README/CLAUDE 상태 표기 |
| ② 프런트 | devpath-frontend, devpath-landing-page | 라우트, 앱별 구현 상태 |
| ③ 인프라·공통 | devpath-shared, devpath-gateway, devpath-gitops, devpath-svc-template | Flyway 마이그레이션 목록, 게이트웨이 라우트, GitOps 앱 등록, 템플릿 표준 |
| ④ 공식 문서 | documents 01~41 | 문서별 "검증 가능한 주장" 추출(구현 상태·엔드포인트·스키마·스택 언급) |
| ⑤ 보조 자산 | storyboard, prototype, workflow-guide, workflow-dashboard, templates | 상태 표기·스택 언급·stale 여부 |

### Phase 3 — 교차 대조 (컨트롤러 직접)
- ④의 문서 주장 × ①②③⑤의 코드 사실을 대조해 불일치 목록 작성
- 각 에이전트 보고에서 핵심 사실 2~3개를 컨트롤러가 직접 재확인(표본 검증)

### Phase 4 — 문서 작성
- 42번 → 43번 순서로 작성, 색인 갱신, 커밋

## 5. 서브에이전트 통제 (scope-lock)

- 지시문에 명시: "이 점검만 수행하라. 끝나면 보고하고 정지하라. 파일을 수정하지 말라. 명세가 부족하면 NEEDS_CONTEXT로 보고하라."
- 전부 읽기 전용 에이전트 사용, 조사 대상 경로·산출 형식을 지시문에 직접 명시
- 보고를 그대로 신뢰하지 않고 표본 검증
- 실패/무응답 시 resume 대신 신규 dispatch (Windows 불안정 이력)

## 6. 브랜치·커밋 전략

- `documents` 레포: `origin/develop` 분기 `docs/consistency-check-round2` (생성 완료)
- 42·43·색인 변경만 커밋 — 다른 세션의 미추적 파일은 건드리지 않음
- develop 대상 PR, CI 녹색 확인 전 머지 금지, main 직접 푸시 금지

## 7. 제외 대상·에러 처리

- 제외: `*.zip`, `*.pdf`, `node_modules`, 빌드 산출물, `.gstack`
- 점검 중 발견되는 버그성 코드 문제는 고치지 않고 42번 "관찰 사항"으로만 기록
- 에이전트가 NEEDS_CONTEXT를 보고하면 컨트롤러가 범위를 보강해 재투입

## 8. 성공 기준

- 42번: 38번 전 항목의 이행 여부가 판정되고, 모든 불일치 항목에 근거(파일 경로·코드 사실)가 붙어 있다
- 43번: 모든 항목이 파일 단위로 실행 가능하고 완료 기준이 검증 가능하다
- 두 문서가 색인에 등록되고 develop 대상 PR로 올라간다
