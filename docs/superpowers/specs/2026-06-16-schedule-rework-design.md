## 1. 배경 & 문제

기존 `17_스케줄.md`는 2026-05-01 착수를 가정한 **24주 순차 계획**(W1 인프라 → W2 OAuth → … → W24 릴리스)이다. 그러나 실제 진행은 계획과 크게 갈라졌다:

- **인프라(W1)**: 전 레포 완료·머지(PostgreSQL 전환, shared GitHub Packages, 중앙 Flyway, 9개 서비스 CI + 이미지/배포 파이프라인 + gitops 자동 SHA).
- **프론트(devpath-frontend)**: React→Flutter 전환 + **목 API 프로토타입**으로 web 골든패스 전체(P4a~f) + admin 3화면(P5)까지 진행. **단, 실서버 연동 0**. 모바일(P6)·랜딩(P7) 미착수.
- **백엔드 도메인**: 9개 서비스 모두 **W1 수준에서 멈춤**(PG 드라이버·shared 의존·DB 연결 테스트). OAuth 이후 도메인 기능 전부 미착수.

즉 "인프라 + 목으로 도는 프론트 데모"는 앞서 있고 **백엔드 도메인이 비어 있다.** 원래의 순차 주차 계획은 현 상태를 반영하지 못한다.

## 2. 목표 & 비목표

**목표**
- 완료된 작업(인프라 + 프론트 목 프로토 + 문서)을 DONE으로 명확히 표기.
- 이후 일정을 외부 마감(모두의 창업 라운드 + 사업계획서 8주 MVP)에 정렬해 재작성.
- 매주 끝단간 동작하는 증분으로 데모 리스크를 분산하고, 솔로 현실을 Tier 컷라인으로 흡수.

**비목표**
- 실제 코드 구현(본 스펙은 일정 문서 재작성까지). 각 슬라이스의 상세 구현은 추후 슬라이스별 브레인스토밍→스펙→TDD.
- workflow-dashboard(17 시드) 재동기화는 별도 도구라 범위 밖(선택 후속).
- 원래 24주 콘텐츠 폐기(버리지 않고 단계에 재배치).

## 6. 수직 슬라이스 순서 & Tier 컷라인

**원칙**: 매 슬라이스 = 백엔드 API + 기존 Flutter 목 화면을 실API로 전환 + 통합테스트(끝단간). 프론트 UI는 이미 존재 → "목을 실데이터로 채움". 매주 데모 가능 증분.

| # | 슬라이스 | 백엔드 | 프론트 전환 | 단계 |
|---|---|---|---|---|
| 1 | OAuth/인증 (게이트) | platform-svc OAuth2(GitHub/Google/카카오)·JWT/Refresh, gateway 엣지, users/oauth/profiles 스키마, UserRegisteredEvent | AuthController 목→실 | MD1 |
| 2 | 진단 | learning-svc question_bank·적응형·비회원 세션(Redis)·결과 이관 | 온보딩 진단 | MD1 |
| 3 | 학습경로 (1st Aha) | ai-svc Claude·Path 프롬프트·SSE, learning-svc pgvector 매칭·milestones/weekly_tasks | PathController SSE | MD1 종착 |
| 4 | 콘텐츠 | learning-svc content·진척 추적 | ContentPage | MD2 |
| 5 | Sandbox | sandbox-svc Docker 풀·gVisor·실행 SSE·sessions | RunController | MD2 |
| 6 | AI 코드리뷰 (2nd Aha) | ai-svc 리뷰 Worker·프롬프트 인젝션 방어·ai_code_reviews | ReviewPanel | MD2 종착 (+결제·베타100) |
| 7 | AI 멘토 | ai-svc mentor·context_snapshot·SSE | MentorController | MD3 |
| 8 | 커뮤니티 Q&A | community-svc Q&A CRUD·AI 시드답변·ES 검색 | 커뮤니티 실연동 | MD3 |
| 9 | LCS (차별화 moat) | Kafka 학습활동 수집·Redis 스냅샷·Post 브릿지·3단계 Sanitize | 맥락 자동첨부 | MD3 |
| 10 | 모바일(P6) | (백엔드 재사용) | apps/mobile 실API·drift·OAuth 딥링크·FCM | MD3 |
| 11 | 랜딩(P7) | — | Jaspr SSG 마케팅 | MD3 |
| 12 | 사업화/완성도 | FinOps(Kill-switch·Semantic Cache)·평판·보안/Chaos·부하/QA | 안정화 | MD4 |

**Tier 컷라인 (솔로 → 우아한 degrade)**
- **Tier-1 (데모 필수)**: #1~6 web 실연동 = MD1+MD2 (1st+2nd Aha). 빠지면 데모 실패.
- **Tier-2 (여유 시, MD3)**: #7 멘토 · #8 커뮤니티 · #9 LCS · #10 모바일 · #11 랜딩.
- **Tier-3 (데모 후/MD4)**: #12 평판 고도화·FinOps 완성·Chaos·부하·AR 등.

**횡단 선결(블로킹)**
- OAuth provider 심사(카카오·Google, 소요 길음)·Anthropic 프로덕션 한도 → **MD1 착수 즉시 신청**.
- 실배포 시크릿(OAuth/Claude 키) 관리, web/admin 실배포 복구(T-DEPLOY-REINTRO — 이미지 파이프라인은 완료).
- 각 슬라이스는 dp_core 실제 시그니처로 플랜 API 가정 사전검증(프론트 P4~P5 반복 교훈), TDD, 브랜치 전략(develop 경유) 준수.
