# 23. DevPath AI 개인정보 영향평가 (자율 PIA)

> **문서 성격**: 개인정보 보호법 제33조에 따른 **법적 의무 영향평가가 아닌** 민간기업의 자율적 개인정보 영향평가. 개인정보위 2025.8. "생성형 AI 개발·활용을 위한 개인정보 처리 안내서" 권장 사항 준수.
>
> **평가 근거**
> - 개인정보보호위원회 『개인정보 영향평가 수행안내서』(공공기관용, 민간 참고) 구조 차용
> - 『생성형 AI 개발·활용을 위한 개인정보 처리 안내서』(2025.8.) 권장사항 반영
> - EU GDPR Article 35 DPIA 프레임워크 일부 차용 (해외 진출 대비)
>
> **법적 효력**: 본 문서는 법적 의무 수행 증빙이 아닌 **자율 평가 기록**이다. 개인정보 침해 사고 발생 시 과태료·과징금 감경 증빙 자료로 제출 가능.
>
> **인센티브**: 개인정보 보호법 위반행위 발생 시 과태료 10% / 과징금 1차 조정금액 30% 감경 (단, 법 제33조제1항 의무 대상은 제외).

## 1. 평가 개요

### 1-1. 평가 목적

- DevPath AI 서비스의 개인정보 처리 과정에서 발생 가능한 개인정보 침해 위험을 **시스템 설계 단계**에서 식별·분석·개선
- 특히 Learning Context Snapshot Service(LCS)의 **에러 로그 처리, AI 답변 생성, 학습 맥락 프로파일링** 세 고위험 영역에 대한 체계적 평가
- Privacy by Design 원칙 준수 증빙
- 향후 유럽 진출 시 GDPR DPIA 재활용 기반 마련

### 1-2. 평가 범위

**포함 (v1.0 MVP)**
- 회원 가입부터 탈퇴까지 전 생명주기
- 학습 시스템·커뮤니티 2개 서브시스템
- LCS (Learning Context Snapshot Service) — 집중 평가
- 제3자 제공 (Anthropic Claude API, OAuth Provider)
- 로그·감사 추적·모니터링 시스템

**제외 (별도 평가)**
- 인사·채용 관련 개인정보 (향후 채용 시 별도 평가)
- 외부 협력사 시스템 내부 처리 (Anthropic 등 — 계약으로 관리)
- **AR 서브시스템**: 본 MVP 제외, v2.0 이후 재활성화 시 별도 평가 (22 참고)
- **결제·유료 플랜**: 본 MVP 제외, 수익화 단계에서 별도 평가 (21 참고)

### 1-3. 평가 수행 체계

| 역할 | 담당 |
|------|------|
| 개인정보보호책임자 (CPO) | 창업자 본인 |
| 평가 수행자 | CPO 직접 수행 |
| 내부 검토 | (기술 고문 위촉 시 자문) |
| 외부 검토 | Phase 2에서 한국인터넷진흥원(KISA) 영향평가 기관 자문 검토 권장 |

### 1-4. 평가 방법론

- 개인정보 흐름도(Data Flow Diagram) 작성 → 흐름별 위험 식별
- 위험 매트릭스 평가 (영향도 5단계 × 발생 가능성 5단계 = 25등급)
- Threat Modeling (STRIDE 프레임워크 일부 차용)
- 개인정보 보호법 시행령 제36조 고려사항 체크

## 2. 서비스 및 개인정보 처리 현황

### 2-1. 서비스 개요

DevPath AI는 **AI 기반 개인화 학습 경로 + Sandbox 실무 AI 코드 리뷰 + 학습 맥락 기반 전문가 커뮤니티**를 결합한 한국 개발자 교육 플랫폼이다. v1.0 MVP는 회원 한정·무료 기반이며, 학습 기록과 커뮤니티 활동을 AI가 분석해 맞춤 경로를 제공한다.

### 2-2. 개인정보 수집·이용 현황

**필수 수집 항목**

| 분류 | 항목 | 수집 시점 | 법적 근거 | 보유 기간 |
|------|------|----------|-----------|-----------|
| 식별정보 | 이메일 주소 | 가입 | 계약 이행 | 탈퇴 시까지 |
| 식별정보 | OAuth 식별자 (GitHub/Google ID) | 가입 | 계약 이행 | 탈퇴 시까지 |
| 인증정보 | 암호화된 비밀번호 | 가입 (이메일 가입 시) | 계약 이행 | 탈퇴 시까지 |
| 프로필 | 닉네임 | 가입 | 계약 이행 | 탈퇴 시까지 |
| 학습정보 | 학습 경로, 진척도, 과제 결과 | 서비스 이용 중 | 계약 이행 | 탈퇴 시까지 |
| 활동정보 | 게시물·답변·투표·평판 점수 | 서비스 이용 중 | 계약 이행 | 탈퇴 시까지 (단, 게시글은 탈퇴 후 익명화하여 보관 가능) |
| 접속기록 | IP 주소, User-Agent, 접속 시각 | 모든 접속 | 통신비밀보호법 | 3개월 |

> **결제정보**: v1.0 MVP 범위 제외. 수익화 단계에서 [21_유료_플랜_참고설계.md](./21_유료_플랜_참고설계.md)와 함께 별도 PIA 확장 항목으로 추가 (전자상거래법 5년 보관, 토스페이먼츠 위탁 등).

**선택 수집 항목 (명시적 동의 필요)**

| 분류 | 항목 | 수집 시점 | 기본값 |
|------|------|----------|--------|
| GitHub 분석 | 공개 저장소 목록, 언어, 커밋 활동 | GitHub 로그인 시 | 사용자가 허용 선택 |
| 학습 맥락 | 현재 콘텐츠, 경로, 태그 | 질문 작성 시 | **ON** |
| ⚠ 에러 로그 | Sandbox 에러 메시지·스택트레이스 | 질문 작성 시 | **OFF** (명시적 동의 필요) |
| 프로필 공개 | 회사·직책 | 프로필 입력 시 | 사용자 입력 시만 |

**민감정보 및 고유식별정보**

- 원칙: **수집하지 않음**
- 주민등록번호·여권번호·운전면허번호·외국인등록번호 수집하지 않음
- 민감정보 범주(의료·사상·범죄경력 등): **수집하지 않음**
- 단, **에러 로그 내 비의도적 포함 가능성 존재** → 이 문서의 핵심 위험 영역 (§5 참고)

### 2-3. 제3자 제공 및 위탁

**제3자 제공**: 없음

**처리 위탁 (v1.0 MVP)**

| 수탁자 | 위탁 내용 | 위치 | 근거 |

| SendGrid / SMTP | 이메일 발송 | 국외 | 위탁 계약 |

> **토스페이먼츠 (결제 위탁)**: 수익화 단계에서 추가 예정 (21 참고). 추가 시 본 PIA 갱신.

**국외 이전 고지 의무**: Anthropic API, SendGrid 등 국외 이전 사실을 개인정보 처리방침에 명시.

### 2-4. 개인정보 처리 시스템

| 시스템 | 역할 | 저장 매체 |
|-------|------|-----------|
| User Service | 계정 관리 | PostgreSQL (암호화) |
| Learning Path Engine | 개인화 추천 | PostgreSQL + pgvector |
| Content CMS | 학습 콘텐츠 | PostgreSQL |
| Assessment & Progress | 진단·진척도 | PostgreSQL |
| Community Services (Post/Answer/Reputation 등) | 게시·평판 | PostgreSQL + Redis |
| **LCS (Learning Context Snapshot)** | 학습 맥락 수집·제공 | PostgreSQL + Redis + Kafka |
| Sandbox Runner | 격리 실행 + 에러 Sanitize | Docker (gVisor) |
| Observability | 로그·메트릭 | Loki, Prometheus, S3 |

### 2-5. 처리 환경

- **배포 환경**: Kubernetes (한국 리전)
- **네트워크**: 외부 트래픽은 WAF 경유, 내부는 mTLS
- **백업**: PostgreSQL Point-in-Time Recovery (7일), 객체 스토리지 버전 관리
- **접근 통제**: OAuth2 + RBAC, 관리자 계정 2FA 필수

## 3. 개인정보 흐름 분석

### 3-1. 수집 흐름

```
[사용자]
    ├─ 가입: 이메일/OAuth → User Service → PostgreSQL
    ├─ 학습: 콘텐츠 열람·완료 → Content Service → Kafka → 여러 Consumer
    ├─ 진단: 답변 입력 → Assessment → PostgreSQL + pgvector
    ├─ 실습: 코드 제출 → Sandbox → 임시 컨테이너 (30일 보관)
    ├─ 커뮤니티: 글·답변 → Post/Answer Service → PostgreSQL + Elasticsearch
    └─ 질문 작성: 학습 맥락 요청 → LCS → Redis → (필요 시) PostgreSQL 커밋

[결제 흐름]: v1.0 MVP 미해당. 수익화 단계에서 추가.
```

### 3-2. 이용 흐름

- **개인화 추천**: PostgreSQL의 학습 기록 → Learning Path Engine → Claude API 호출 → 결과 저장
- **AI 답변**: 사용자 질문 → AI Mentor → Claude API → 응답 반환 (대화는 로그로 30일 보관)
- **커뮤니티 답변**: LCS에서 맥락 조회 → 답변자에게 노출 → 답변 작성
- **분석**: 익명화 집계 통계 → 내부 분석 (개인 식별 불가)

### 3-3. 제공·위탁 흐름

- **Claude API 전송 시 최소화 원칙**: 이메일·이름 등 식별정보는 전송 안 함. 유저 ID는 해시 후 전송.
- **결제 데이터 흐름**: v1.0 미해당 (수익화 단계에서 추가 — PG 직접 POST, DevPath AI 서버 경유 안 함)

### 3-4. 보관 및 파기 흐름

| 데이터 | 보관 위치 | 보관 기간 | 파기 방법 |
|-------|----------|----------|----------|
| 계정 정보 | PostgreSQL | 탈퇴 시까지 | 물리적 삭제 + 백업 30일 내 삭제 |
| 학습 기록 | PostgreSQL | 탈퇴 시까지 | 물리적 삭제 |
| 커뮤니티 글 | PostgreSQL | 탈퇴 후 6개월 | 익명화 (공익·사회적 가치 고려) |
| LCS 스냅샷 | PostgreSQL | 첨부된 질문 생명주기 | 게시물 삭제 시 cascade |
| 실시간 캐시 | Redis | TTL 7일 | 자동 만료 |
| 원본 에러 로그 | Sandbox DB | 30일 | 자동 삭제 |
| Sanitized 에러 | 스냅샷에 포함 | 질문 생명주기 | cascade |
| 접속 기록 | Loki + S3 | 3개월 | 자동 삭제 |
| Claude API 요청 로그 | 내부 + Anthropic ZDR | 30일 / 0일 (ZDR) | Anthropic 정책 |

> 결제 내역 보관은 수익화 시 전자상거래법 5년 적용 (21 참고).

## 4. 위험 식별 및 평가

### 4-1. 위험 평가 매트릭스

**영향도**
- 5 (매우 높음): 대규모 민감정보 유출, 형사 책임
- 4 (높음): 개인정보 유출, 행정 처분
- 3 (보통): 개별 사용자 피해
- 2 (낮음): 불편, 신뢰 저하
- 1 (매우 낮음): 경미

**발생 가능성**
- 5 (매우 높음): 일상적 발생
- 4 (높음): 월 1회 이상
- 3 (보통): 연 1~2회
- 2 (낮음): 드물게
- 1 (매우 낮음): 예외적

**위험 등급 = 영향도 × 가능성**
- 20~25: Critical (즉시 대응)
- 12~16: High (30일 내 대응)
- 6~10: Medium (계획적 대응)
- 1~5: Low (모니터링)

### 4-2. 식별된 위험 목록 (v1.0 MVP 범위)

| ID | 위험 시나리오 | 영향 | 가능성 | 등급 | 영역 |
|----|-------------|-----|--------|------|------|
| **R-01** | **에러 로그 Sanitize 누락으로 민감정보 노출** | 5 | 4 | **Critical (20)** | LCS |
| **R-02** | **AI 응답에 학습 맥락 내 개인정보가 포함되어 재생산** | 4 | 3 | **High (12)** | AI Mentor |
| **R-03** | **답변자가 학습 맥락을 악용해 프로파일링** | 4 | 2 | **Medium (8)** | Community |
| R-04 | 내부자 DB 직접 접근으로 대량 열람 | 5 | 2 | High (10) | 관리 |
| R-05 | SQL Injection·권한 우회 외부 공격 | 5 | 2 | High (10) | 보안 |
| R-06 | 백업 파일 유출 | 4 | 2 | Medium (8) | 운영 |
| R-07 | OAuth 토큰 탈취 | 4 | 3 | High (12) | 인증 |

### 4-3. Critical 위험 상세 (R-01)

**시나리오**: 사용자가 Sandbox에서 에러를 경험했을 때 에러 메시지에 포함된 민감정보(이메일, API 키, DB URL, 환경변수 등)가 Sanitize 파이프라인을 통과하지 못하고 스냅샷에 첨부되어 답변자·외부에 노출되는 상황.

**왜 Critical인가**:
- 에러 로그는 환경변수·설정파일·코드를 담고 있어 민감정보가 섞일 확률이 높음
- 한 번 스냅샷으로 commit되면 불변 → 사후 수정 어려움
- 답변자(타인)가 즉시 볼 수 있음 → 유출 즉시 확정

**대응 (§6 참고)**: 3-Stage Sanitize 파이프라인 + 기본 OFF + Fail-Safe 원칙 + AI 검증 + 정기 Red Team 테스트

### 4-4. High 위험 상세 (R-02, R-07, R-15, R-19, R-20)

**R-02: AI 응답에 학습 맥락 내 개인정보 포함 후 재생산**
- 사용자가 AI 멘토에게 "내 XXX를 분석해줘"로 본의 아니게 개인정보 포함 → Claude 응답에 그대로 노출 → 로그에 저장
- 대응: Claude API 전송 전 입력 필터, 응답 후 출력 필터 (2025.8. AI 안내서 권장)

**R-07: OAuth 토큰 탈취**
- GitHub/Google 토큰 유출 시 공격자가 타인 계정 로그인 가능
- 대응: 토큰 암호화 저장, 정기 갱신, 이상 접속 탐지

**R-15: 로그에 민감정보 저장**
- 개발자가 무심코 로그 print에 개인정보 남김
- 대응: 로그 Sanitizer 미들웨어, 로그 정책 가이드, 배포 전 자동 스캔

**R-19: LCS 스냅샷 무제한 공개**
- 답변자 권한 체크가 부실해 무관한 유저가 스냅샷 조회
- 대응: `visibility` 필드 엄격 체크, Audit trail, Rate limit

**R-20: 국외 이전 미고지**
- Claude API 이용 자체가 국외 이전 → 개인정보 처리방침에 명시 필수
- 대응: 처리방침 고지, 동의 화면 포함, Anthropic과 위탁 계약 체결

## 5. LCS 집중 심층 평가

LCS는 DevPath AI의 **가장 고위험 컴포넌트**이므로 별도 심층 평가.

### 5-1. LCS의 특수성

| 특성 | 일반 시스템 대비 리스크 |
|------|----------------------|
| 에러 로그 처리 | 민감정보 포함 가능성 극대 |
| 타인 노출 | 답변자가 정보 열람 → 즉시 공개 확정 |
| 불변성 | Commit 후 수정 불가 → 사후 대응 제한 |
| 자동화 | 사용자 주의 없이 자동 수집 |
| 다중 필드 조합 | 여러 필드가 합쳐져 개인 식별력 증가 |

### 5-2. LCS 위험 세부 분석

**필드별 민감도 등급**

| 필드 | 민감도 | 기본값 | 근거 |
|-----|--------|--------|------|
| current_content | Low | ON | 서비스 내 콘텐츠 ID |
| current_path | Low | ON | 학습 경로 ID |
| active_tags | Low | ON | 태그는 일반 용어 |
| tag_reputation | Low | ON | 이미 프로필 공개 |
| level | Low | ON | 이미 프로필 공개 |
| **recent_errors** | **High** | **OFF** | **환경변수·경로·토큰 포함 가능** |

### 5-3. LCS에 적용되는 Privacy by Design 원칙

**1. Proactive not Reactive (선제적 대응)**
- Sanitize 파이프라인을 처리 시점이 아니라 **수집 시점**에 적용
- 에러 로그는 원본을 LCS로 가져오지 않음, Sandbox Service에서 Sanitize 후 전달

**2. Privacy as the Default (기본값 보호)**
- `recent_errors` 필드 기본 OFF
- 기타 필드도 visibility 기본값 `answerers_only`

**3. Privacy Embedded into Design (내재화)**
- Sanitize 파이프라인은 우회 불가 (아키텍처 수준 강제)
- 모든 스냅샷 조회는 권한 체크 경유 필수

**4. Full Functionality (기능 손실 없음)**
- Sanitize가 정보를 완전히 제거하는 게 아니라 **마스킹** — 답변자가 문맥은 이해 가능
- "[DB_URL]"이라는 마스킹이 "MySQL에 연결 못함" 맥락을 유지

**5. End-to-End Security**
- 수집 시 TLS 1.3, 저장 시 AES-256-GCM, 전송 시 mTLS

**6. Visibility and Transparency**
- 사용자는 자신의 스냅샷 모두 조회 가능
- 누가 자신의 스냅샷을 봤는지 Audit trail 조회 가능
- Sanitize 룰 목록 공개 (이런 패턴을 마스킹한다고 명시)

**7. Respect for User Privacy**
- 필드별 on/off 세밀한 제어
- 언제든 스냅샷 삭제 가능
- 기능 전면 거부 옵션

### 5-4. LCS 사고 시나리오 및 대응 시뮬레이션

**시나리오 A: Sanitize 버그로 DB URL 유출**

가정: 정규식 룰에 버그, `mysql://admin:pw@...` 형태 일부가 마스킹되지 않음.

즉시 대응:
1. Feature flag로 `recent_errors` 수집 전면 중단 (15분 이내)
2. 지난 7일 생성된 스냅샷 자동 재검증 스크립트 실행
3. 영향받은 스냅샷 자동 삭제 또는 재Sanitize
4. 영향받은 사용자 개별 통지 (이메일)
5. 개인정보위 신고 여부 판단 (유출 규모 1,000건 이상 시 신고)

재발 방지:
- 정규식 룰 단위 테스트 강화
- Stage 3 AI 검증 샘플링률 일시 상향 (5% → 20%)
- Red Team 테스트 정기화 (분기 1회)

**시나리오 B: 답변자의 악성 프로파일링 시도**

가정: 한 답변자가 짧은 시간 내 수십 명의 스냅샷을 조회.

자동 탐지:
- 답변자별 시간당 스냅샷 조회 수 > 20 → 자동 알림
- 한 답변자가 여러 질문자의 `recent_errors` 필드만 집중 조회 → 이상 패턴

조치:
- 자동 Rate limit 강화
- 관리자 검토 → 악의 확인 시 계정 정지
- 피해 가능성 있는 질문자에게 통지

### 5-5. LCS 개인정보 최소 수집 원칙

LCS는 `answer quality`를 높이는 것이 목적이므로, 실제로 답변 품질에 기여하는 최소한의 정보만 수집.

**수집하지 않는 것** (명시):
- 사용자 실명·이메일·전화번호
- IP 주소·디바이스 정보
- 다른 사용자와의 대화 내용
- 학습 기록의 상세 답변 텍스트 (정답/오답 결과만)
- 결제·구독 정보

## 8. AI 특화 평가 항목

개인정보위 2025.8. "생성형 AI 개발·활용을 위한 개인정보 처리 안내서" 권장 평가 기준에 따른 항목별 점검.

### 8-1. 목적 설정 단계

| 항목 | 준수 여부 | 비고 |
|------|:--------:|------|

| 개인정보 이용 필수성 검토 | ✓ | 익명 사용자에게도 기본 기능 제공 |

## 11. 사고 대응 플레이북

### 11-1. 사고 심각도 분류

| 등급 | 정의 | 예시 |
|------|------|------|
| P1 (Critical) | 대규모 유출, 민감정보 포함 | 1,000건 이상 유출 / 금융 정보 포함 |
| P2 (High) | 중규모 유출 또는 시스템 침해 | 100~1,000건 유출 |
| P3 (Medium) | 소규모, 제한적 | 10~100건 유출 또는 내부 오접근 |
| P4 (Low) | 개별 사건 | 1~10건 |

### 11-2. 단계별 대응

**Phase 1: 탐지 및 즉시 대응 (0~1시간)**
1. 사고 탐지 (자동 알림 또는 신고)
2. CPO 즉시 투입
3. 피해 확산 차단 (feature flag, 서비스 격리)
4. 증거 보존 (로그 스냅샷)

**Phase 2: 영향 파악 (1~24시간)**
5. 유출 범위 확인 (누구·무엇·얼마나)
6. 원인 분석
7. 영향받은 사용자 식별
8. 법적 고지 의무 판단

**Phase 3: 사용자 및 당국 고지 (24~72시간)**
9. 영향받은 사용자 개별 통지
10. 개인정보위 신고 (1,000건 이상 또는 민감정보 포함 시 필수)
11. 홈페이지 공지
12. 언론 대응 (필요 시)

**Phase 4: 복구 및 재발 방지 (1주일 이내)**
13. 기술적 결함 수정
14. 프로세스 개선
15. 사후 보고서 작성
16. PIA 재검토

### 11-3. 고지 템플릿

**사용자 개별 통지 (이메일)**
```
제목: [중요] DevPath AI 개인정보 관련 중요 안내

안녕하세요, [이름]님.

DevPath AI에서 개인정보 처리와 관련하여 중요한 사항을 안내드립니다.

[사고 내용 요약]
- 발생 일시: YYYY-MM-DD HH:MM
- 영향 범위: [귀하의 어떤 정보가 영향받았는지]
- 원인: [간결한 설명]

[대응 조치]
- 이미 취한 조치
- 앞으로 할 조치

[사용자께 권장드리는 조치]
- 비밀번호 변경 등

[문의처]
- 개인정보보호책임자: [연락처]
- 개인정보위 신고 센터: privacy.go.kr
```

**개인정보위 신고 (72시간 내)**
- 신고 양식: privacy.go.kr 제공
- 필수 항목: 유출 항목·규모·원인·대응·재발 방지책

### 11-4. 법정 대응 준비

- 법률 자문 연락처 미리 확보
- 보험: 개인정보 유출 배상책임보험 가입 검토
- 기록: 모든 대응 기록 보존 (증거)
