# 12. 코드 리뷰 규칙

> **목표**: 결함 조기 발견, 지식 공유, 일관된 품질, 심리적 안전성
> **원칙**: 코드는 공유 자산. 사람이 아닌 코드를 비평한다.

---

## 1. 역할과 책임

### 1.1 작성자(Author)
- PR 제출 전 self-review
- 변경 의도 명확히 서술 (템플릿)
- 리뷰 피드백에 **24시간 이내 응답** (근무일)
- 피드백에 동의/대안/근거 중 하나로 답변

### 1.2 리뷰어(Reviewer)
- **48시간 이내 첫 리뷰** (근무일)
- 가치 있는 피드백 우선, 스타일은 Lint에 위임
- 필요 시 화상 짝 리뷰

### 1.3 Code Owner
- 모듈 아키텍처 일관성 보증
- 고위험 모듈(Sandbox, OAuth)은 필수 리뷰어 지정

---

## 2. 주목 포인트 (What Gets Reviewed)

| 영역 | 주목 포인트 |
|------|------------|
| 도메인 모델 | 불변식, Aggregate 경계 |
| OAuth2 | 토큰 저장 암호화, 스코프 최소화, 콜백 검증 |
| **Sandbox** | **격리 정책, 리소스 제한, 탈출 가능성** |
| AI 프롬프트 | 버전 관리, 비용, PII |
| 이벤트/Outbox | dedup_key, 멱등, DLQ |
| 성능 | N+1, 인덱스, 캐시 일관성 |
| 프런트 | 접근성, i18n, LCP/CLS |

## 3. 리뷰 체크리스트

### 3.1 공통
- [ ] PR 제목/본문이 의도를 전달
- [ ] 변경 라인 400↓, 파일 10↓
- [ ] Conventional Commits
- [ ] 문서 갱신 여부 확인
- [ ] Feature Flag 필요성
- [ ] 테스트 추가/수정

### 3.2 도메인
- [ ] 불변식 생성자/팩토리/명령 경계 강제
- [ ] 불변식 위반 테스트
- [ ] 유비쿼터스 언어 일치
- [ ] Aggregate 간 ID 참조
- [ ] 순환 의존 없음

### 3.3 데이터
- [ ] Flyway 이름 컨벤션 `V{date_seq}__{snake}.sql`
- [ ] 인덱스 EXPLAIN 첨부
- [ ] Zero-downtime 전략
- [ ] 롤백 스크립트 또는 전략

### 3.4 API
- [ ] 하위 호환 (필드 삭제는 deprecation)
- [ ] 에러 코드 일관성
- [ ] `@PreAuthorize` 적절
- [ ] Rate Limit 고려
- [ ] OpenAPI 자동 문서 반영

### 3.5 보안 (OAuth/Sandbox)
- [ ] 사용자 입력 신뢰 금지
- [ ] OAuth 토큰 암호화 저장 (AES-256-GCM)
- [ ] OAuth 스코프 최소 권한
- [ ] JWT 검증 누락 없음
- [ ] **Sandbox 리소스 제한** 코드 내 명시
- [ ] **Sandbox 네트워크 차단** 확인
- [ ] 비밀키 하드코딩 없음

### 3.6 성능
- [ ] N+1 방지 (fetch join / EntityGraph / QueryDSL)
- [ ] 인덱스 사용
- [ ] 페이지네이션
- [ ] 계측 없이 캐시 추가 금지
- [ ] SSE backpressure

### 3.7 AI / LLM
- [ ] 프롬프트 버전 관리
- [ ] Semantic Cache 적용
- [ ] FinOps 단가 추적 (`ai_cost_logs`)
- [ ] Output PII 스캔
- [ ] Temperature / max_tokens 근거

### 3.8 Sandbox (전용)
- [ ] gVisor 런타임 명시
- [ ] seccomp 프로파일
- [ ] readonly rootfs
- [ ] 네트워크 정책
- [ ] 컨테이너 1회 사용 후 파기
- [ ] 한도 체크 (일/월)

### 3.9 프런트엔드
- [ ] 접근성 (ARIA, 키보드, 대비)
- [ ] i18n (하드코딩 없음)
- [ ] 상태 관리 일관성
- [ ] 리렌더 비용 (memoization 근거)
- [ ] 에러 바운더리

### 3.10 테스트
- [ ] Happy path + 엣지
- [ ] Mock 적절 (도메인은 실객체)
- [ ] Flaky 방지 (시계 주입, Awaitility)
- [ ] 커버리지 감소 시 근거

## 4. 승인 기준

**Approve 가능 조건**:
1. CI 모든 필수 체크 통과
2. 체크리스트 관련 항목 확인
3. 남은 피드백은 "nit"뿐
4. 아키텍처/보안/성능 의문 없음

**Request changes**:
- 설계 의문
- 보안/데이터 정합성 리스크
- 테스트 누락 명백
- 하위 호환 깨짐

**Comment만**:
- 추가 정보 필요
- 대안 제시 후 의견 대기

---

## 5. 피드백 커뮤니케이션

### 5.1 언어
- 주체는 코드, 사람이 아님
- 근거 덧붙임
- 대안 제시

### 5.2 접두어

| 접두어 | 의미 | 예 |
|--------|------|-----|
| `nit:` | 사소 | `nit: 변수명 fetchDueCards` |
| `suggestion:` | 제안 | `suggestion: CompletableFuture 병렬화` |
| `question:` | 이해 | `question: 이 분기 도달 가능한가?` |
| `issue:` | 수정 필요 | `issue: Sandbox timeout이 제대로 안 걸림` |
| `blocker:` | 반드시 수정 | `blocker: OAuth 토큰 평문 저장됨` |
| `praise:` | 칭찬 | `praise: Chaos 테스트까지 ⭐` |

### 5.3 토론 종결
- 의견 갈릴 때: 결정권자 지정 (Author or Code Owner)
- 오프라인 30분 제한
- 결정은 PR 코멘트로 기록

---

## 6. PR 크기 기준

| 변경 라인 | 정책 |
|----------|------|
| < 400줄 | 일반 리뷰 (1인 승인) |
| 400~600줄 | **시니어 리뷰 필수** (2인 승인) |
| > 600줄 | **분할 필수** — Stacked PR 권장 |

- 불가피한 대형 PR: 커밋 단위 논리 분리 / 본문에 리뷰 순서 명시 / 화상 선행 리뷰

---

## 7. 보안 리뷰

- OAuth, Sandbox 격리 → **보안 팀 리뷰어 추가**
- 비밀값 커밋 시:
  1. 즉시 토큰 revoke
  2. PR close
  3. `git filter-repo`
  4. 사건 공유

---

## 8. AI 보조 도구

- Sourcery / Copilot PR Summary 등은 **보조**
- AI 제안은 Author가 이해 후 수락

---

## 9. 리뷰 지표 (월간)

| 지표 | 목표 |
|------|------|
| 1차 리뷰 응답 중앙값 | < 4시간 |
| 머지까지 총 시간 중앙값 | < 24시간 |
| PR 평균 크기 | < 400 라인 |
| 머지 후 24h 롤백률 | < 2% |
