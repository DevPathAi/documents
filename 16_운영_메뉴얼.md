# 16. 운영 매뉴얼

> **대상**: SRE · 운영자 · 관리자
> **범위**: 모니터링, FinOps, Sandbox 보안, 장애 대응

---

## 1. 모니터링 스택

| 구성 | 도구 | 용도 |
|------|------|------|
| 메트릭 | Prometheus + Grafana | 시스템/AI/FinOps/Sandbox |
| 로그 | Loki (또는 ELK) | 구조화 JSON |
| 트레이싱 | OpenTelemetry + Zipkin | 분산 추적 |
| 알람 | Alertmanager → Slack + PagerDuty | |
| APM | Micrometer | JVM, HTTP, DB |

### 1.1 대시보드 목록

1. **System** — 노드 CPU/메모리/네트워크, K8s
2. **Application** — 엔드포인트 p50/p95/p99, 5xx, QPS
3. **Database** — MySQL/pgvector/Redis/ES
4. **Kafka** — Consumer Lag, DLQ
5. **AI Quality** — 코드 리뷰 👍 / 멘토 👍 / 프롬프트 A/B
6. **FinOps** — 서비스별 비용, cache hit, kill-switch
7. **2 Aha 퍼널** — 가입→1st Aha→2nd Aha 전환율
8. **Sandbox Operations** — 풀 사용률/대기 큐/악용 플래그
9. **OAuth/GitHub** — 콜백 성공률, GitHub API Rate Limit

## 2. 2 Aha 퍼널 대시보드 (핵심)

```text
Funnel Monitor
├── 방문자 → 비회원 진단 시작 (conversion %)
├── 진단 시작 → 완료 (%)
├── 진단 완료 → 가입 (로그인 게이트 통과율)
├── 가입 → 1st Aha (경로 수신) 도달 시간 p50/p95
├── 1st Aha → 첫 콘텐츠 진입 (%)
├── 첫 콘텐츠 → Sandbox 제출 (%)
└── Sandbox 제출 → AI 리뷰 수신 (2nd Aha) (%)

알람:
- 1st Aha 도달률 < 60%
- 가입 → 1st Aha p50 > 10분
- 2nd Aha D0 도달률 < 35%
```

**배포 직후 15분 관찰 트리거**: 이 퍼널의 변화가 자동 롤백 조건.

## 3. 주요 SLO 메트릭

| 메트릭 | 임계 |
|--------|------|

## 4. 알람 규칙 (발췌)

```yaml
groups:
- name: slo
  rules:
  - alert: ApiP95High
    expr: histogram_quantile(0.95, sum(rate(http_request_duration_seconds_bucket[5m])) by (le,route)) > 2.5
    for: 10m
    labels: { severity: high }

- name: aha-funnel
  rules:
  - alert: FirstAhaRateDrop
    expr: sum(rate(path_generated_total[15m])) / sum(rate(user_registered_total[15m])) < 0.6
    for: 15m
    labels: { severity: critical }
    annotations: { summary: "1st Aha 도달률 60% 미만 — 배포 직후면 롤백 검토" }

- name: sandbox
  rules:
  - alert: SandboxPoolExhaustion
    expr: sandbox_queue_depth > 30
    for: 5m
    labels: { severity: warning }
  - alert: SandboxAbusePattern
    expr: rate(sandbox_abuse_flagged_total[10m]) > 0.5
    labels: { severity: critical }

- name: oauth
  rules:
  - alert: OAuthCallbackFailureSpike
    expr: sum(rate(oauth_callback_total{status="failure"}[5m])) / sum(rate(oauth_callback_total[5m])) > 0.05
    labels: { severity: critical }

- name: finops
  rules:
  - alert: DailyCostSoftLimitReached
    expr: increase(ai_cost_usd_total[1d]) > ${DAILY_COST_SOFT_LIMIT_USD}
    labels: { severity: medium }
```

## 5. FinOps 운영

### 5.1 Kill-switch 3계층

| 레벨 | 기준 | 조치 |
|------|------|------|
| 사용자별 | 월 한도 초과 | 해당 사용자 AI 호출만 차단 (콘텐츠 열람 유지) |
| 이상 패턴 | 5분 내 100회 이상 | 자동 Rate Limit 강화 |
| 전사 | Hard limit | 전면 Kill + PagerDuty |

### 5.2 단가 지표

| 지표 | 목표 |
|------|------|
| cost_per_path_generation | < $0.03 |
| cost_per_code_review | < $0.01 |
| cost_per_mentor_message | < $0.005 |
| cost_per_active_learner | < $0.30 |
| semantic_cache_hit_rate | > 40% |

### 5.3 튜닝 포인트

- 모델 라우팅: 짧은 멘토 응답은 Haiku (비용 1/5)
- Semantic Cache TTL (기본 7일)
- 이상 호출 패턴 자동 차단

## 6. Sandbox 보안 운영

### 6.1 악용 탐지 지표

| 지표 | 대응 |
|------|------|
| 사용자 CPU 100% 3회 이상/일 | 경고 배너 → 5회 이상 시 Sandbox 일시 정지 |
| 30초 timeout 반복 | 같은 사용자 10회/일 시 사용자 확인 요청 |
| 다량 PID 생성 시도 | 즉시 kill + 플래그 |

### 6.2 보안 점검 루틴

- 주간: Sandbox pentest CI job 재실행 + 결과 리뷰
- 월간: gVisor 버전 업데이트 + seccomp 프로파일 감사
- 분기: 외부 침투 테스트 (Sandbox 탈출 시도)

### 6.3 장애 시 Fallback

- Sandbox 풀 고갈: 503 + `Retry-After` 30초
- gVisor 장애: 일시적 runsc → runc 전환은 **금지** (보안 저하). 차라리 Sandbox 기능 Off.

## 7. OAuth · GitHub API 운영

### 7.1 GitHub Rate Limit

- 사용자 토큰 기준 5,000/hr per user
- Worker는 사용자별로 throttle
- Rate Limit 근접 시 refresh 대기

### 7.2 OAuth 토큰 회전

- Access Token 만료: Spring Security 자동 갱신
- Client Secret 월 1회 회전 (Runbook §12)
- 토큰 암호화 키 분기 회전 (dual-write → re-encrypt)

### 7.3 GitHub API 장애

- Worker 지연 → 경로 생성은 GitHub 없이 진행 가능 (진단만으로)
- 복구 시 재수집 배치

2. OpenAI fallback 확인
3. 멘토는 fallback 메시지, Path 생성은 재시도 큐

1. HPA 스케일 확인
2. 대기 큐 사이즈
3. 악용 패턴 사용자 격리

1. 특정 provider인지 확인 (GitHub/Google/카카오)

3. Redirect URI 설정 재확인

### 9.4 2 Aha 퍼널 하락
1. Funnel 각 단계 분석
2. 최근 배포 혐의 → Canary 롤백
3. Path 생성 실패 로그 확인

## 13. 백업 · 복원 절차

### 13.1 자동 백업
| 대상 | 방식 | 주기 | 보존 기간 |
|------|------|------|----------|
| MySQL (SSOT) | RDS 자동 스냅샷 | 매일 03:00 KST | 30일 |
| PostgreSQL (pgvector) | pg_dump + S3 업로드 | 매일 04:00 KST | 30일 |
| Redis | RDB 스냅샷 | 매 6시간 | 7일 |
| Elasticsearch | Snapshot to S3 | 매일 05:00 KST | 14일 |
| Kafka | Topic retention | 7일 (자동) | — |

### 13.2 수동 복원
1. RDS 콘솔 → Point-in-Time Recovery (최대 5분 단위)
2. pgvector: `pg_restore` from S3
3. Redis: `redis-cli --rdb` 또는 AOF 재생
4. ES: `POST /_snapshot/s3_repo/snap_name/_restore`

### 13.3 백업 검증
- 월 1회 staging에서 복원 드릴 실행
- 복원 후 데이터 정합성 체크 스크립트 실행 (users 수, posts 수 비교)
