## 1. 결론

현재 정합성은 **핵심 방향은 일치하지만, 구현 현황 표기는 아직 모두 맞춰지지 않았다**.

- 제품 방향, 타깃, 2 Aha Moment, LCS 차별화, Flutter 전환, PostgreSQL/Flyway 중심 구조는 문서·프로토타입·스토리보드·레포가 대체로 같은 방향을 가리킨다.
- 실제 구현은 **MD1 일부**까지 진행되어 있다. 백엔드는 `platform-svc` 인증/사용자, `learning-svc` 온보딩 진단/비회원 진단/claim, `ai-svc` Ollama 게이트웨이가 확인된다.
- 많은 상위 문서는 v1 목표 상태를 설명하고 있다. 그러나 일부 README/워크플로/일정표는 “미착수” 또는 “Claude 전용”처럼 현재 코드와 다른 표현을 유지한다.
- `prototype`과 `storyboard`는 실제 백엔드 연동 제품이 아니라 **정적/목 데이터 기반 데모와 화면 설계**로 정리되어 있어 이 부분은 정합하다.
- 가장 큰 불일치는 `17_스케줄.md`, 레포별 `docs/project-management/workflow/*.md`, `devpath-ai-svc` README/CLAUDE, `workflow-guide`의 과거 MySQL 문서다.

따라서 “모두 맞춰졌는가?”에 대한 답은 **아니오**다. 다만 불일치 대부분은 설계 방향 충돌이 아니라 **현재 구현/목표/과거 계획 라벨링 문제**다.

실제 공식 구조는 PostgreSQL 17 + pgvector + Redis + Kafka + Elasticsearch다. 이 가이드는 현재 제품 문서라기보다 과거/공용 가이드 성격으로 보이지만, 전체 작업공간 기준으로는 불일치다.

권장 조치:
- `workflow-guide`를 현행 프로젝트 가이드로 유지할지, 아카이브로 둘지 결정한다.
- 유지한다면 MySQL을 PostgreSQL로 교체한다.

### F8. 레포별 워크플로 체크박스가 실제 코드보다 느림

레포별 `docs/project-management/workflow/*.md`는 아직 다수 항목이 미완료다. 일부는 실제 미구현이라 맞지만, 일부는 이미 구현된 항목을 반영하지 못한다.

예:
- `WORKFLOW_platform_MD1.md`의 JWT/Refresh/UserRegisteredEvent 관련 항목
- `WORKFLOW_learning_MD1.md`의 question_bank/assessment/guest claim 관련 항목
- `WORKFLOW_ai_MD1.md`의 Claude 경로 생성 항목은 현재 Ollama로 방향이 바뀌었음

권장 조치:
- 구현 상태를 체크박스 하나로만 표현하지 말고 `done / partial / planned / superseded` 상태를 도입한다.

### F9. `documents/Home.md`의 스케줄 설명 과거 버전

점검 중 `Home.md`의 `17_스케줄` 설명이 “24주 Phase 1~6 (M1~M6)”로 남아 있음을 확인했다. 실제 `17_스케줄.md`는 모두의 창업 라운드 정렬 기반 MD1~MD4 문서다.

조치:
- 본 점검과 함께 Home의 `17_스케줄` 설명을 MD1~MD4 기준으로 변경했다.
