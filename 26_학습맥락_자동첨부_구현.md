# 학습 맥락 자동 첨부 시스템 · 기술 구현 상세

> **문서 목적**: DevPath AI의 핵심 차별화 기능인 "학습 맥락 자동 첨부"의 엔지니어링 구현 스펙.
>
> **문서 성격**: v1 목표 구현 스펙. 현재 `devpath-lcs-svc`는 스냅샷 draft/commit/조회 2종·preferences GET/PUT 6개 엔드포인트와 `learning_context_snapshots`·`user_context_preferences` 스키마가 구현되어 있으며, `sanitize_rules` 테이블과 이벤트 스트림 수집(9개 Kafka 토픽)은 TARGET(미구현)이다. 아래 API 경로는 이미 실제 구현 prefix(`/lcs/*`)로 정정되어 있다(작성 당시 계획값 `/api/lcs/*`였던 것을 실제 구현에 맞춰 교체함). 전체 구현 현황은 [42_전체_정합성_점검_2차](./42_전체_정합성_점검_2차.md)를 기준으로 확인한다.
>
> **대상 독자**: 백엔드·DevOps 엔지니어. 이 문서만 보고 구현 시작 가능한 수준의 상세도.
>
> **핵심 설계 결정**:
> - Snapshot 생성 방식: **Hybrid** (실시간 캐시 + 요청 시 조립)
> - Snapshot 생명주기: **Dual** (첨부본 영속 · 캐시 휘발)
> - 프라이버시 기본값: **세분화** (항목별 민감도 차등)
> - 에러 수집: **Sanitize 파이프라인 필수**
>
> **관련 서비스**: Learning Context Snapshot Service (기존 아키텍처의 Bridge Component)

## 1. 시스템 개요

### 1-1. 책임 범위

Learning Context Snapshot Service(이하 **LCS**)는 다음 책임을 가진다.

- **수집**: 학습 시스템의 이벤트 스트림을 구독해 유저별 최신 학습 상태를 유지
- **Sanitize**: 에러 로그에서 민감 정보를 자동 마스킹
- **조립**: 요청 시 유저의 "현재 맥락 스냅샷"을 생성
- **영속**: 질문에 첨부된 스냅샷을 불변 기록으로 저장
- **프라이버시**: 유저 선호에 따라 필드 수집·노출 제어

### 1-2. 명시적 비책임

혼선 방지를 위해 LCS가 **하지 않는** 것도 명시한다.

- 콘텐츠 메타데이터 직접 관리 (Content CMS 소관)
- 답변자 UI 렌더링 (Community UI 소관)
- 질문 본문 저장 (Post Service 소관)
- 장기 학습 분석 (Analytics Service 별도)

LCS는 얇은 브릿지 서비스다.

### 1-3. 핵심 요구사항

| 요구사항 | 값 |
|---------|-----|
| Snapshot 생성 latency (P95) | < 100ms |
| Snapshot 조립 성공률 | > 99.5% |
| 실시간 캐시 신선도 (staleness) | < 5초 |
| 민감정보 유출률 | 0% (목표) |
| 저장 데이터 보존 | 질문 생명주기 = 게시물 삭제 시 함께 삭제 |

### 2-1. 플로우의 핵심 특성

**Write Path**는 이벤트 소싱 기반 + 비동기로, 학습 시스템에 부하를 주지 않는다. 이벤트 소비가 밀려도 질문 작성은 blocking되지 않는다 (최대 5초 staleness 허용).

**Read Path**는 Redis 우선 조회 + 필요 시 동기 fallback. 99%는 Redis hit으로 끝나고, miss 시에만 학습 서비스에 쿼리한다.

**Dual lifecycle**: 질문 미리보기용 임시 snapshot은 TTL 10분 Redis에, 질문이 실제 게시되면 PostgreSQL에 영속화. 게시 안 된 미리보기는 자동 소멸.

## 3. Snapshot 생성 전략

### 3-1. Hybrid 모델

Snapshot은 다음 3단계로 조립된다.

**Stage 1: 실시간 캐시 조회 (빠른 경로)**
```
GET redis: user:{userId}:context
→ JSON 역직렬화 → Snapshot 객체
```
일반적으로 < 5ms.

**Stage 2: Fallback 조회 (캐시 miss 또는 stale)**
Redis에 없거나 `last_updated_at`이 10분 이전이면:
```
parallel:
  - Content Service: GET /api/content/users/{userId}/current
  - Assessment Service: GET /api/assessment/users/{userId}/latest
  - Sandbox Service: GET /api/sandbox/users/{userId}/errors?limit=3
join results → Snapshot
async: Redis 업데이트
```
P95 ~ 60ms (세 서비스 병렬 호출).

**Stage 3: 프라이버시 필터 적용**
```
user_preferences = GET redis: user:{userId}:privacy
for field in snapshot:
    if user_preferences[field] == false:
        remove field
```

### 3-2. 왜 Hybrid인가

순수 Pull은 매번 3개 서비스 호출 → UX 지연. 순수 Push는 이벤트 누락 시 영구 불일치. Hybrid는 **"최선 노력 실시간 + 필요 시 정확성 보정"** 의 타협이다.

### 3-3. Staleness 관리

Redis 엔트리에 `last_updated_at` 필드 포함. 현재 시각과 비교해:
- < 1분: 바로 사용
- 1~10분: 사용하되 백그라운드로 비동기 refresh
- > 10분 또는 miss: Stage 2 fallback 강제

이렇게 하면 UX는 거의 항상 빠르고, 정확성은 점진적으로 수렴한다.

### 3-4. 일관성 보장 원칙

**중요**: Snapshot은 "이 순간의 유저 상태"를 정확히 반영할 필요가 없다. 답변자에게 유용한 **근사값**이면 충분하다. 이 인식이 설계 단순화의 핵심이다.

강제 일관성이 필요한 경우(예: 평판 점수)는 다른 서비스가 담당한다. LCS는 "대략 맞는 맥락"만 제공하면 된다.

## 4. 이벤트 스트림 수집

### 4-1. 구독 토픽 목록

LCS Event Consumer가 구독하는 Kafka 토픽.

| 토픽 | 발행 주체 | 이벤트 예시 |
|------|----------|------------|
| `learning.content.opened` | Content Service | 유저가 콘텐츠 페이지 진입 |
| `learning.content.progressed` | Content Service | 스크롤/시청 진척도 변경 |
| `learning.content.completed` | Content Service | 콘텐츠 완료 |
| `learning.path.assigned` | Learning Path Engine | 새 학습 경로 할당 |
| `learning.path.step_changed` | Assessment Service | 경로 내 현재 단계 변경 |
| `learning.assessment.completed` | Assessment Service | 진단·과제 완료 |
| `learning.sandbox.error` | Sandbox Runner | 실습 환경 에러 발생 |
| `learning.sandbox.completed` | Sandbox Runner | 실습 과제 완료 |
| `community.post.created` | Post Service | (역방향) 질문 발행 시 snapshot commit |

### 4-2. 이벤트 스키마 표준

모든 이벤트는 공통 헤더를 갖는다.

```json

  "event_type": "learning.content.opened",

  "user_id": "usr_01HXXXX",
  "correlation_id": "corr_01HXXXX",

  "payload": { /* 이벤트별 내용 */ }
}
```

**주요 이벤트 payload 예시**

`learning.content.opened`
```json
{
  "content_id": "content_jpa_nplus1",
  "path_id": "path_backend_3m",
  "step_number": 1,
```

`learning.sandbox.error`
```json

  "sandbox_session_id": "sbx_01HXXXX",
  "exercise_id": "ex_jpa_01",
  "error_type": "LazyInitializationException",
  "error_message_raw": "[원본 메시지, sanitize 대상]",
  "stack_trace_raw": "[...]",

```

### 4-3. Consumer 구현 (Spring Boot + Spring Kafka)

**핵심 코드 패턴** (의사 코드)

```java
@Component
public class LearningContextConsumer {

    private final RedisTemplate<String, String> redis;
    private final ErrorSanitizer sanitizer;
    private final ObjectMapper mapper;

    @KafkaListener(
        topics = {
            "learning.content.opened",
            "learning.content.progressed",
            "learning.content.completed",
            "learning.path.step_changed"
        },
        groupId = "lcs-context-updater",
        concurrency = "3"
    )
    public void handleContentEvent(ConsumerRecord<String, String> record) {
        try {
            LearningEvent event = mapper.readValue(record.value(), LearningEvent.class);
            String cacheKey = "user:" + event.getUserId() + ":context";
            
            // 현재 캐시 읽기 → 병합 → 쓰기 (Lua 스크립트로 atomic하게)
            UserContext current = fetchOrInit(cacheKey);
            current.applyEvent(event);
            current.setLastUpdatedAt(Instant.now());
            
            redis.opsForValue().set(
                cacheKey,
                mapper.writeValueAsString(current),
                Duration.ofDays(7)
            );
            
            // 관측
            metrics.counter("lcs.events.processed", "type", event.getType()).increment();
            
        } catch (Exception e) {
            // DLQ로 보냄. 중단 금지
            dlq.send(record.value(), e);
            metrics.counter("lcs.events.failed", "type", event.getType()).increment();
        }
    }

    @KafkaListener(
        topics = "learning.sandbox.error",
        groupId = "lcs-error-collector"
    )
    public void handleSandboxError(ConsumerRecord<String, String> record) {
        SandboxErrorEvent event = mapper.readValue(record.value(), SandboxErrorEvent.class);
        
        // Sanitize 파이프라인 통과
        SanitizedError sanitized = sanitizer.sanitize(event);
        
        // 민감정보 검출 시 마스킹된 버전만 저장
        String cacheKey = "user:" + event.getUserId() + ":recent_errors";
        redis.opsForList().leftPush(cacheKey, mapper.writeValueAsString(sanitized));
        redis.opsForList().trim(cacheKey, 0, 9); // 최근 10개만
        redis.expire(cacheKey, Duration.ofDays(7));
    }
}
```

### 4-4. Idempotency 처리

같은 이벤트가 두 번 처리되어도 결과가 같아야 한다.

- Consumer 측에서 `event_id`를 Redis Set(`processed_events`)에 기록, TTL 1시간
- 이미 있으면 skip
- 이벤트 갱신은 "마지막 값 승리(LWW)" + `timestamp` 비교로 out-of-order 이벤트 방지

### 4-5. 백프레셔 및 장애 격리

- Consumer group 병렬도: 토픽별 3 (파티션 수만큼 확장 가능)
- DLQ (Dead Letter Queue): 처리 실패 3회 시 `lcs.dlq.events` 토픽으로 격리
- Circuit Breaker: Redis 연결 불가 시 이벤트 처리 중단, 학습 서비스는 계속 발행 (backlog는 Kafka가 유지)
- Lag 알림: Consumer lag이 1,000 이벤트 초과 시 Slack 알림

## 5. 에러 로그 수집 및 Sanitize 파이프라인

가장 민감한 파트입니다. 이 섹션의 구현이 프로덕션 안전성을 결정합니다.

### 5-1. 수집 대상

Sandbox Runner에서 실습 중 발생한 에러. 기본적으로 다음이 포함된다.

- Exception class name (ex. `LazyInitializationException`)
- Error message (1st line)
- Stack trace (top 5 frames)
- 관련 파일·라인 번호 (있으면)
- Timestamp

**수집하지 않는 것** (명시적 제외):
- 환경 변수 전체 덤프
- Database connection string
- HTTP request body
- 인증 토큰·API 키
- 시스템 파일 경로 (`/home/유저명/...`)

### 5-2. Sanitize 파이프라인 3단계

```
raw error log
      │
      ▼
┌──────────────────────────┐
│ Stage 1: 구조화 파싱     │
│ stack trace 분리         │
│ 불필요 라인 제거         │
└──────────┬───────────────┘
           │
           ▼
┌──────────────────────────┐
│ Stage 2: 정규식 마스킹   │
│ 알려진 민감 패턴 치환    │
└──────────┬───────────────┘
           │
           ▼
┌──────────────────────────┐
│ Stage 3: AI 검증 (옵션)  │
│ Claude로 의심 필드 탐지  │
│ 고비용이라 샘플링만      │
└──────────┬───────────────┘
           │
           ▼
  Sanitized Error
```

### 5-3. Stage 2: 정규식 마스킹 룰 (필수)

프로덕션에 반드시 적용해야 하는 패턴들.

| 패턴 | 정규식 | 치환 |
|------|--------|------|
| 이메일 | `[\w.+-]+@[\w-]+\.[\w.-]+` | `[EMAIL]` |
| 한국 주민번호 | `\d{6}-[1-4]\d{6}` | `[RRN]` |
| 전화번호 | `01[016789]-?\d{3,4}-?\d{4}` | `[PHONE]` |
| 신용카드 | `\b(?:\d{4}[- ]?){3}\d{4}\b` | `[CARD]` |
| API Key (generic) | `(api[_-]?key\|token\|secret)[=:\s]+["']?[\w\-]{20,}` | `[API_KEY]` |
| AWS Access Key | `AKIA[0-9A-Z]{16}` | `[AWS_KEY]` |
| JWT | `eyJ[\w-]+\.[\w-]+\.[\w-]+` | `[JWT]` |
| IPv4 | `\b(?:\d{1,3}\.){3}\d{1,3}\b` | `[IP]` |
| 홈 디렉토리 경로 | `/home/[\w]+/\|C:\\Users\\[\w]+\\` | `/home/[USER]/` |
| DB URL | `(jdbc\|mysql\|postgres)://[\w:]+@[\w.]+` | `[DB_URL]` |
| URL with query params | `\?[\w&=]+` | `?[QUERY]` |

**한국 특화 패턴**은 추가로 정리:
- 사업자번호 (10자리), 여권번호, 운전면허번호

**구현 전략**: 룰 테이블을 DB에 관리, 핫 리로드 가능하게. 새 민감 패턴 발견 시 코드 배포 없이 추가.

### 5-4. Stage 3: AI 검증 (옵션, 샘플링)

정규식으로 잡히지 않는 케이스 (예: "password is hunter2" 같은 자연어).

**구현 방식**:
- Stage 2 통과한 에러 중 **5% 샘플링**
- Claude Haiku로 검증: "이 텍스트에 민감정보가 있는지 JSON으로 답하라"
- 있으면 플래그 + 관리자 큐에 적재 + 정규식 룰 후보로 제안
- 비용: 샘플 5%만 → 월 X만원 수준

**주의**: AI에 에러 로그를 보내는 행위 자체가 프라이버시 이슈. Anthropic의 Zero Data Retention 모드 + Amazon Bedrock 경유로 데이터 주권 확보. 유저 ID는 해시 후 전송.

### 5-5. Sanitize 결과 포맷

```json
{
  "error_type": "LazyInitializationException",
  "error_message": "failed to lazily initialize a collection of role: Order.items",
  "stack_frames": [
    "at com.example.[APP_PKG].OrderService.findAll(OrderService.java:42)",
    "at com.example.[APP_PKG].OrderController.list(OrderController.java:28)"
  ],
  "file_context": "OrderService.java:42",
  "sanitize_meta": {
    "patterns_matched": ["file_path"],
    "version": "2026-04-22",
    "ai_verified": false
  },
  "original_hash": "sha256:abc123...",
  "created_at": "2026-04-22T10:32:15Z"
}
```

`original_hash`를 두는 이유: 같은 에러 반복 탐지 (중복 제거).

### 5-6. 유저의 원본 접근

유저는 **자신의 원본 에러 로그를 볼 수 있어야** 한다 (디버깅 목적). 그러나 Snapshot에 첨부되는 것은 Sanitized 버전만.

- 원본: Sandbox Session 데이터베이스에 암호화 저장, 30일 후 자동 삭제
- 첨부본: Sanitized 버전, 질문 생명주기만큼 유지

### 5-7. Fail-Safe 원칙

**Sanitizer가 실패하면 에러 로그는 첨부되지 않는다.** 첨부 실패가 민감정보 유출보다 안전하다.

```java
try {
    SanitizedError result = sanitizer.sanitize(raw);
    snapshot.setErrorContext(result);
} catch (SanitizerException e) {
    // 로그 첨부 포기, 다른 맥락은 유지
    metrics.counter("lcs.sanitize.skipped").increment();
    alertIfRepeated(e);
}
```

```sql
CREATE TABLE learning_context_snapshots (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    
    -- 영속 목적
    purpose VARCHAR(32) NOT NULL,  -- 'question_attachment', 'analytics'
    attached_to_type VARCHAR(32),   -- 'question', 'answer'
    attached_to_id UUID,            -- 게시물 ID
    
    -- 실제 맥락 데이터 (불변)
    content_snapshot JSONB NOT NULL,
    
    -- 프라이버시 메타
    visibility VARCHAR(16) NOT NULL DEFAULT 'answerers_only',  -- 'public', 'answerers_only', 'private'
    fields_included TEXT[] NOT NULL,  -- ['current_content', 'recent_errors', ...]
    
    -- 감사
    created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    sanitize_version VARCHAR(32),
    
    -- 제약
    CHECK (purpose IN ('question_attachment', 'analytics')),
    CHECK (visibility IN ('public', 'answerers_only', 'private'))
);

CREATE INDEX idx_lcs_user_id ON learning_context_snapshots(user_id);
CREATE INDEX idx_lcs_attached ON learning_context_snapshots(attached_to_type, attached_to_id);
CREATE INDEX idx_lcs_created_at ON learning_context_snapshots(created_at DESC);

-- Privacy preferences (user별)
CREATE TABLE user_context_preferences (
    user_id UUID PRIMARY KEY REFERENCES users(id) ON DELETE CASCADE,
    
    -- 필드별 기본 수집 허용
    collect_current_content BOOLEAN NOT NULL DEFAULT TRUE,
    collect_learning_path BOOLEAN NOT NULL DEFAULT TRUE,
    collect_active_tags BOOLEAN NOT NULL DEFAULT TRUE,
    collect_recent_errors BOOLEAN NOT NULL DEFAULT FALSE,  -- ← 기본 OFF
    collect_tag_reputation BOOLEAN NOT NULL DEFAULT TRUE,
    collect_level BOOLEAN NOT NULL DEFAULT TRUE,
    
    -- 노출 범위 기본값
    default_visibility VARCHAR(16) NOT NULL DEFAULT 'answerers_only',
    
    updated_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

-- Sanitize rules (hot-reloadable)
CREATE TABLE sanitize_rules (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    rule_name VARCHAR(64) NOT NULL,
    pattern TEXT NOT NULL,        -- 정규식
    replacement VARCHAR(64) NOT NULL,
    priority INT NOT NULL DEFAULT 100,
    is_active BOOLEAN NOT NULL DEFAULT TRUE,
    created_at TIMESTAMPTZ DEFAULT now(),
    updated_at TIMESTAMPTZ DEFAULT now()
);

CREATE INDEX idx_sanitize_active ON sanitize_rules(is_active, priority);
```

### 6-3. Snapshot 불변성

질문에 첨부된 snapshot은 **한 번 저장되면 수정 불가**. 이유:
- 답변자가 본 맥락이 사후 변경되면 신뢰가 깨짐
- 감사 추적 용이
- 프라이버시 정책 변경 시에도 과거 snapshot은 당시 정책 적용

유저가 "이 snapshot 삭제"를 원할 때만 physical delete. 이 경우 질문에는 "맥락이 삭제됨" 표시.

### 6-4. 자동 삭제 정책

- 질문 삭제 시 → 첨부된 snapshot cascade delete (PostgreSQL ON DELETE CASCADE)
- 유저 탈퇴 시 → 해당 유저의 모든 snapshot delete
- Analytics 목적 snapshot → 90일 후 익명화 (user_id 해시)

### 8-3. GDPR / 개인정보보호법 대응

**권리 보장**

| 권리 | 구현 |
|------|------|
| 열람권 (Right to Access) | `GET /lcs/snapshots?user_id=me` — 본인의 모든 snapshot 조회 |
| 정정권 (Right to Rectification) | Snapshot은 불변 → "삭제 후 재생성" 안내 |
| 삭제권 (Right to Erasure) | `DELETE /lcs/snapshots/{id}` → physical delete |
| 처리 제한권 | Privacy preferences 전면 off 기능 |
| 데이터 이동성 | JSON export (기존 snapshot들 포함) |
| 반대권 | 기능 전면 거부 가능 (질문은 맥락 없이 작성됨) |

**데이터 처리 근거 (한국 개인정보보호법 기준)**
- 수집·이용: 유저 동의 (제15조 1항 1호)
- 파기: 목적 달성 즉시 또는 유저 요청 시
- 제3자 제공: 없음 (답변자도 같은 서비스 내)
- 국외 이전: Sanitize 시 Claude API 경유 시 고지 필수

### 8-4. 암호화

- **저장 시**: PostgreSQL TDE(Transparent Data Encryption) + `content_snapshot` JSONB는 추가로 애플리케이션 레벨 암호화 (AES-256-GCM)
- **전송 시**: 내부 서비스 간 mTLS, 외부는 TLS 1.3
- **Redis**: TLS + AUTH, 민감 필드는 애플리케이션 레벨 암호화

암호화 키 관리: AWS KMS 또는 HashiCorp Vault.

## 9. 성능 및 비용

### 9-1. 성능 목표 (재확인)

| 작업 | P50 | P95 | P99 |
|------|-----|-----|-----|
| Draft snapshot 생성 (cache hit) | 15ms | 40ms | 80ms |
| Draft snapshot 생성 (cache miss) | 50ms | 120ms | 250ms |
| Commit snapshot | 30ms | 80ms | 150ms |
| Read snapshot (답변자) | 10ms | 25ms | 50ms |
| Sanitize (정규식 only) | 2ms | 8ms | 20ms |
| Sanitize (AI 검증 포함) | 800ms | 2000ms | 4000ms |

### 9-2. 예상 트래픽 (MVP 기준)

- DAU: 1,000명
- 질문 작성: 일 100건
- Draft snapshot 호출: 일 300건 (질문당 평균 3회 미리보기)
- Snapshot 조회: 일 1,500건 (질문당 답변자 평균 15명 조회)
- Sandbox 에러 이벤트: 일 500건

### 9-3. 인프라 리소스 추정

**LCS 서비스**
- Kubernetes Pods: 3 replicas (HA)
- Pod spec: 500m CPU / 512Mi RAM (과소할당 안 해도 충분)
- Auto-scaling: CPU 70% 시 최대 10 replicas

**Redis**
- Memory: 유저당 평균 5KB (context + errors) × 10K 유저 = 50MB
- 실사용: Redis 인스턴스 1GB면 여유 (DAU 10배 성장까지)

**PostgreSQL**
- Snapshot 로우당 평균 2KB
- 질문 1년 × 일 100건 × 2KB = **약 73MB / 년**
- 실질적으로 무시할 수준

**Kafka**
- 이벤트 수: 일 50만 건 (DAU 1K 기준)
- 파티션: 토픽당 3개
- Retention: 7일 (백필 대비)

### 9-4. Claude API 비용 (AI Sanitize 옵션)

- Stage 3 샘플링 5%: 일 25건 (sandbox.error 500건 × 5%)
- 건당 ~500 input tokens, ~50 output tokens
- Haiku 단가 기준: 일 수십 원 수준 → 월 1~2만원

매우 미미함. 켜서 무방.

| Sanitize 실패율 > 1% | Critical | 에러 수집 즉시 중단, 페이지 콜 |
| P95 latency > 500ms | Warning | 원인 조사 |
| 403 비율 > 5% | Warning | 권한 로직 버그 의심 |
| 프라이버시 거부율 급증 | Info | UX 팀에 알림 (동의 문구 개선 여지) |

## 11. Failure Modes 및 대응

### 11-1. 시나리오별 대응

**Scenario 1: Redis 전체 장애**
- 영향: 실시간 캐시 miss → 모든 요청이 fallback pull로 직행 → 학습 서비스 부하 증가, 지연 5배
- 대응: Circuit breaker가 학습 서비스 보호, LCS는 "맥락 일시 불가" 응답
- UX: "학습 맥락을 불러올 수 없습니다. 맥락 없이 질문하거나 잠시 후 재시도"

**Scenario 2: Content Service 장애**
- 영향: `current_content` 필드가 fallback도 실패
- 대응: 부분 snapshot 반환 (다른 필드는 정상)
- UX: "현재 학습 콘텐츠 정보를 가져오지 못했습니다"라고 명시, 나머지는 첨부

**Scenario 3: Kafka Consumer Lag 폭증**
- 영향: Redis 캐시가 점점 stale해짐
- 대응: Staleness 감지 시 자동 fallback으로 전환, Lag 알림
- UX: 영향 없음 (fallback이 동작)

**Scenario 4: Sanitize 파이프라인 버그로 민감정보 유출**
- 영향: 치명적
- 대응:
  1. Feature flag로 `recent_errors` 수집 즉시 전면 중단
  2. 최근 1주일 내 해당 snapshot 자동 재처리 또는 삭제
  3. 영향받은 유저 공지 및 조치
  4. Postmortem 작성

**Scenario 5: 대량 삭제 요청 (GDPR 유사 사태)**
- 영향: 순간적으로 DB 부하
- 대응: 삭제 작업은 `DELETE FROM ... WHERE id IN (...)` 배치 처리 + 백그라운드 Job 큐
- UX: "삭제 요청이 접수되었습니다. 48시간 내 처리" 고지

### 11-2. 데이터 복구

질문에 첨부된 snapshot이 잘못 삭제되었을 때:
- PostgreSQL Point-in-time Recovery (PITR) → 7일 이내 복구 가능
- 이것도 안 되면 원본 이벤트로부터 재조립 시도 (Kafka 7일 retention)

### 11-3. 로컬 대안 (Degraded Mode)

최악의 경우, LCS 서비스 전체가 다운되어도:
- Community는 LCS 없이 질문 작성 가능 (맥락 없음)
- Community UI는 LCS API 타임아웃 500ms 설정, 실패 시 "맥락 불가" 표시 후 질문 플로우 계속 진행

**LCS는 UX 개선이지 필수가 아니다.**

### Sanitizer 클래스 (Spring Boot)

```java
@Service
public class ErrorSanitizer {
    
    private final SanitizeRuleRepository ruleRepo;
    private final ClaudeClient claudeClient;
    private final MeterRegistry metrics;
    
    @Cacheable(value = "sanitize-rules", unless = "#result.isEmpty()")
    public List<SanitizeRule> loadActiveRules() {
        return ruleRepo.findAllActive();
    }
    
    public SanitizedError sanitize(SandboxErrorEvent event) {
        String original = event.getErrorMessageRaw();
        String originalHash = DigestUtils.sha256Hex(original);
        
        // Stage 1: 구조화 파싱
        ParsedError parsed = parse(event);
        
        // Stage 2: 정규식 마스킹
        List<String> matched = new ArrayList<>();
        String sanitized = parsed.getMessage();
        for (SanitizeRule rule : loadActiveRules()) {
            if (Pattern.compile(rule.getPattern()).matcher(sanitized).find()) {
                sanitized = sanitized.replaceAll(rule.getPattern(), rule.getReplacement());
                matched.add(rule.getRuleName());
                metrics.counter("lcs.sanitize.patterns_matched", "pattern", rule.getRuleName()).increment();
            }
        }
        
        // Stage 3: AI 검증 (5% 샘플링)
        boolean aiVerified = false;
        if (Math.random() < 0.05) {
            try {
                boolean hasSensitive = claudeClient.checkSensitive(sanitized);
                if (hasSensitive) {
                    // 알림 + 관리자 큐
                    alertService.notify("Sanitize Stage 3 detected leak: " + originalHash);
                    throw new SanitizerException("AI detected residual sensitive data");
                }
                aiVerified = true;
            } catch (ClaudeApiException e) {
                // AI 검증 실패는 non-blocking
                metrics.counter("lcs.sanitize.ai_skipped").increment();
            }
        }
        
        return SanitizedError.builder()
            .errorType(parsed.getErrorType())
            .errorMessage(sanitized)
            .stackFrames(parsed.getTopFrames(5))
            .patternsMatched(matched)
            .aiVerified(aiVerified)
            .originalHash(originalHash)
            .sanitizeVersion("2026-04-22")
            .build();
    }
}
```

### Snapshot Builder

```java
@Service
public class SnapshotBuilder {

    private final UserContextCache cache;
    private final ContentServiceClient contentClient;
    private final AssessmentServiceClient assessmentClient;
    private final SandboxServiceClient sandboxClient;
    private final PrivacyService privacyService;
    
    public DraftSnapshot buildDraft(UUID userId, List<String> requestedFields) {
        // 1. 프라이버시 preferences
        UserPreferences prefs = privacyService.getPreferences(userId);
        
        // 2. Redis 우선
        Optional<UserContext> cached = cache.get(userId);
        
        UserContext context;
        if (cached.isPresent() && cached.get().isFresh()) {
            context = cached.get();
        } else {
            // 3. Parallel fallback
            context = pullInParallel(userId);
            cache.putAsync(userId, context);
        }
        
        // 4. 필드별 프라이버시 필터
        Map<String, Object> filteredContent = new HashMap<>();
        List<FieldStatus> fieldStatuses = new ArrayList<>();
        
        for (String field : requestedFields) {
            if (!prefs.allows(field)) {
                fieldStatuses.add(FieldStatus.denied(field, "user_preference_off"));
                continue;
            }
            Object value = context.getField(field);
            if (value == null) {
                fieldStatuses.add(FieldStatus.unavailable(field, "no_data"));
                continue;
            }
            filteredContent.put(field, value);
            fieldStatuses.add(FieldStatus.included(field));
        }
        
        // 5. Draft 저장 (Redis, TTL 10분)
        String draftId = generateUlid();
        DraftSnapshot draft = DraftSnapshot.builder()
            .id(draftId)
            .userId(userId)
            .content(filteredContent)
            .fieldStatuses(fieldStatuses)
            .expiresAt(Instant.now().plus(Duration.ofMinutes(10)))
            .build();
        cache.putDraft(draftId, draft);
        
        return draft;
    }
    
    private UserContext pullInParallel(UUID userId) {
        CompletableFuture<ContentSnapshot> content = 
            CompletableFuture.supplyAsync(() -> contentClient.getCurrent(userId));
        CompletableFuture<AssessmentSnapshot> assessment = 
            CompletableFuture.supplyAsync(() -> assessmentClient.getLatest(userId));
        CompletableFuture<List<SanitizedError>> errors = 
            CompletableFuture.supplyAsync(() -> sandboxClient.getRecentErrors(userId, 3));
        
        try {
            CompletableFuture.allOf(content, assessment, errors).get(500, TimeUnit.MILLISECONDS);
        } catch (Exception e) {
            // Partial success 허용
            log.warn("Partial pull: {}", e.getMessage());
        }
        
        return UserContext.combine(
            content.getNow(null),
            assessment.getNow(null),
            errors.getNow(Collections.emptyList())
        );
    }
}
```
