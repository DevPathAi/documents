# 14. 배포 가이드

> **환경**: dev → staging → production
> **오케스트레이션**: Kubernetes (EKS/GKE) + Helm
> **CI/CD**: GitHub Actions + ArgoCD (선택)
> **배포 전략**: Feature Flag + Rolling / Canary (Production)

---

## 1. 배포 아키텍처

```text
GitHub (main) ──▶ GitHub Actions ──▶ Container Registry (ECR/GCR)
                                         │
                                         ▼
                            Helm + ArgoCD / kubectl
                                         │
                    ┌────────────────────┼────────────────────┐
                    ▼                    ▼                    ▼
                 dev (EKS)          staging (EKS)         prod (EKS)
                 Single AZ          Single AZ             Multi-AZ HA
```

## 2. 환경별 정책

| 항목 | dev | staging | production |
|------|-----|---------|-----------|
| 브랜치 | feature | main | release/v* 태그 |
| 승인 | 자동 | 자동 | **수동 승인 2건** |
| 데이터 | 샘플 | 마스킹 스냅샷 | 실데이터 |
| LLM | 개인 키 | staging 키 | 운영 키 |
| Sampling | 100% | 50% | 10~30% |
| 배포 전략 | Rolling | Rolling | Canary → Rolling |
| Sandbox 풀 | 5 pods | 20 pods | HPA (20~100) |

## 3. CI 파이프라인

```text
.github/workflows/
├── ci.yml                # PR / push 빌드 + 테스트
├── security.yml          # 주간 Snyk/Trivy + Sandbox pentest
├── ar-test.yml           # Firebase Test Lab (릴리즈 전) — (v2.0 — 현재 비활성)
├── deploy-staging.yml    # main push → staging
└── deploy-prod.yml       # tag push → prod (수동 승인)
```

### 3.1 PR CI 흐름

1. **Lint**: ktlint, flutter analyze, dart format
2. **Unit**: JUnit, flutter test
3. **Integration**: Testcontainers
4. **Security**: Snyk, detect-secrets, Trivy
5. **Sandbox pentest**: 격리 탈출 시도 5종
6. **Build**: Gradle, flutter build
7. **Quality Gate**: SonarQube
8. **Image**: 브랜치별 태깅

### 3.2 Main → Staging 자동

```yaml
on:
  push:
    branches: [main]
jobs:
  build-push:
    steps:
      - uses: docker/build-push-action@v5
        with:
          tags: |
            ${{ env.REGISTRY }}/devpath-api:${{ github.sha }}
            ${{ env.REGISTRY }}/devpath-api:staging-latest
  deploy:
    needs: build-push
    steps:
      - name: Helm upgrade
        run: |
          helm upgrade --install devpath-api ./charts/devpath-api \
            --namespace devpath-staging \
            --set image.tag=${{ github.sha }} \
            --values values-staging.yaml
      - name: Smoke Test
        run: flutter test integration_test --tags smoke
```

### 3.3 Prod (태그 기반)

```yaml
on:
  push:
    tags: ['v*']
jobs:
  approval:
    environment: production
  deploy-canary:
    steps:
      - run: helm upgrade ... --set canary.weight=10
      - name: Observe 15min
        run: sleep 900
      - name: Check SLO
        run: ./scripts/check-slo.sh
      - name: Check 2 Aha funnel
        run: ./scripts/check-aha-funnel.sh
  deploy-full:
    steps:
      - run: helm upgrade ... --set canary.weight=100
```


### 4.1 표준 관례

- **Probes**: readiness / liveness / startup
- **Resources**: requests + limits 명시
- **HPA**: CPU 60% + Kafka lag (custom)
- **PDB**: minAvailable 1
- **PodAntiAffinity**: 동일 노드 회피

### 4.2 Sandbox Runner 특수 설정

```yaml
# charts/devpath-sandbox-runner/templates/deployment.yaml
spec:
  template:
    spec:
      runtimeClassName: gvisor  # runsc 런타임
      containers:
        - name: runner
          securityContext:
            readOnlyRootFilesystem: true
            allowPrivilegeEscalation: false
            capabilities:
              drop: [ALL]
          resources:
            limits:
              cpu: "1"
              memory: 512Mi
          volumeMounts:
            - mountPath: /workspace
              name: tmp
      volumes:
        - name: tmp
          emptyDir:
            medium: Memory
            sizeLimit: 128Mi
---
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
spec:
  minReplicas: 20
  maxReplicas: 100
  metrics:
    - type: External
      external:
        metric:
          name: sandbox_queue_depth
        target:
          type: AverageValue
          averageValue: "5"
```

### 5.1 Feature Flag

- **기본 On 이후 Feature Flag 제거** 원칙 (v1.0 이후)
- 신규 기능은 Flag 뒤에:
  - `ai-mentor-context-injection`: 기본 On

  - `guest-assessment`: 기본 On

### 5.2 Canary (Production)

1. **Canary 10%**: 이미지 교체 (Istio VirtualService)
2. **15분 관찰**: p95, 5xx, **1st Aha 도달률**
3. **SLO 이탈 자동 롤백**
4. **통과**: 50% → 100%

### 5.3 Blue-Green

- 스키마 변경 등 고위험 시

## 6. DB 마이그레이션

### 6.1 Flyway

- 모든 DDL/DML은 Flyway
- 파일명: `V{yyyyMMdd_seq}__{description}.sql`
- 원칙: forward-only

### 6.2 Zero-Downtime

| 변경 | 전략 |
|------|------|
| 컬럼 추가 (nullable) | 단일 배포 |
| 컬럼 추가 (NOT NULL) | 3단계: ADD + BACKFILL + ALTER |
| 컬럼 삭제 | 2단계: 코드 제거 → 다음 릴리즈에 DROP |
| 인덱스 추가 | `CREATE INDEX CONCURRENTLY` |
| OAuth 토큰 암호화 키 회전 | dual-write + re-encrypt 배치 |

## 7. 롤백 절차

### 7.1 앱 롤백
```bash
helm rollback devpath-api <revision> -n devpath-prod
```

### 7.2 마이그레이션 롤백
- Forward-only → revert 마이그레이션 작성
- 대안: Feature Flag Off (데이터 유지)

### 7.3 Canary 자동 롤백 트리거
- p95 SLO 이탈 3회 연속
- 5xx > 0.5% (5분)
- **1st Aha 도달률 50% 이하 (15분)** ← Aha 퍼널이 배포 품질 시그널
- Consumer lag > 60s

## 10. 장애 대응

### 10.1 Severity
| Severity | 기준 | 초동 |
|----------|------|------|
| SEV-1 | 전사 장애, 데이터 손실 | 즉시 온콜 + 롤백 판단 |
| SEV-2 | 주요 기능 장애 | 30분 내 |
| SEV-3 | 일부 기능 열화 | 영업시간 |

### 10.2 플레이북
1. Slack `#incident` 스레드 (Commander 지정)
2. 영향 범위 공표
3. 원인 가설 → 롤백 or 핫픽스
4. 정상화 후 **Post-mortem 72h 내**

## 12. DR (Disaster Recovery) 절차

### 12.1 RTO / RPO 목표
| 구분 | 목표 |
|------|------|
| RTO (Recovery Time Objective) | ≤ 1시간 |
| RPO (Recovery Point Objective) | ≤ 15분 |

### 12.2 시나리오별 대응
| 장애 유형 | 대응 | 예상 복구 시간 |
|----------|------|--------------|
| DB 인스턴스 장애 | RDS Multi-AZ 자동 페일오버 | ~5분 |
| AZ 장애 | K8s 노드 자동 스케줄링 | ~10분 |
| 리전 장애 | DNS 전환 + 백업 리전 복구 | ~60분 |
| 데이터 오염 | Point-in-Time Recovery (PITR) | ~30분 |
| Kafka 클러스터 장애 | Outbox 재발행 (last_published_at 기준) | ~15분 |

### 12.3 DR 훈련
- 분기 1회 DR 드릴 실시 (staging 환경)
- 훈련 후 Post-mortem 작성 및 RTO/RPO 실측값 기록

## 13. 롤백 후 검증 체크리스트

롤백 실행 후 반드시 아래 항목을 순서대로 확인한다.

- [ ] 헬스체크 엔드포인트 200 응답 (`/actuator/health`)
- [ ] DB 마이그레이션 버전 일치 (`flyway_schema_history` 확인)
- [ ] OAuth 로그인 정상 (GitHub/Google/카카오 각 1회)
- [ ] 학습 경로 생성 SSE 스트리밍 정상
- [ ] Sandbox 실행 → AI 코드 리뷰 수신 정상
- [ ] 커뮤니티 Q&A 작성 → AI 시드 답변 생성 정상
- [ ] Kafka Consumer lag < 10초
- [ ] 5xx 에러율 < 0.1% (5분 관찰)
- [ ] 1st Aha 퍼널 도달률 정상 범위 (15분 관찰)
- [ ] Grafana 대시보드 이상 알림 없음
