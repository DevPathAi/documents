## 1. 수정 원칙

1. **현재 구현과 목표 설계를 분리한다.** 코드·마이그레이션·라우트로 확인된 범위만 구현으로 표기하고, 목표는 TARGET, 시점 스냅샷·과거 계획은 HISTORY, 목 데모는 PROTOTYPE으로 라벨한다.
2. **완료를 과장하지 않는다.** 실측·실행 이력이 없는 결과 수치는 남기지 않고, E2E가 아닌 것은 부분 구현으로 표기한다.
3. **문서 소비자가 먼저 보는 파일부터 고친다.** 일정표 → 레포 README → 규칙 문서 → 설계 문서 배너 → 과거 문서 순.
4. **"현재 기준" 참조는 42번으로 통일한다.** 36·37번을 현재 기준으로 참조하는 배너는 42번 참조로 교체한다(36·37은 HISTORY로 보존).

## 7. 실행 규칙

- 각 항목은 해당 레포의 `docs/*`(또는 `fix/*`) 브랜치에서 수행하고 develop 대상 PR로 머지한다(merge commit, CI 녹색 확인).
- documents 항목(P0-1·2·3·5, P1-1~8·14, P2-1·2)은 한 브랜치로 묶어 진행해도 된다 — 상호 독립적이므로 항목 단위 커밋을 유지한다.
- 서비스 레포 README 항목(P0-4·6, P1-9~13)은 레포별 별도 브랜치·PR.
- P1-15(.github)는 웹 렌더링 확인이 선행 — 확인 결과에 따라 수정 여부 결정.
- P0-5의 "사용자 공통 규칙(git-branch-flow)"은 레포 외부의 팀 운영 규칙이다. 실행자가 규칙 전문에 접근할 수 없으면 완료 기준의 문구 부재 판정("develop 사용 안 함"·"Merge Commit 금지" 제거)만으로 판정한다.
- 각 항목의 완료 판정은 표의 "완료 기준"으로 하며, 완료 후 42번의 해당 M 항목에 이행 표기를 남긴다.
