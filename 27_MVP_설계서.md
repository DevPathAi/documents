## 1️⃣ MVP 정의: 8주 후 무엇이 존재하는가

### 1-1. 핵심 정의

**MVP**(Minimum Viable Product)는 "최소한의 기능으로 검증 가능한 제품"이다. 흔히 오해하는 두 가지를 명확히:

❌ MVP는 "완성된 제품의 90%"가 아니다.
❌ MVP는 "기능을 최대한 많이 넣은 베타"가 아니다.

✅ **MVP는 "이 사업의 핵심 가설을 검증하기 위한 최소 단위의 제품"**이다.

### 1-2. DevPath AI MVP의 범위

8주 알파에 **반드시 들어가는 것**:

| 기능 | 이유 |
|------|------|
| 15문항 적응형 진단 테스트 | "수준 진단 → 맞춤 로드맵" 가설 검증의 출발점 |
| AI 기반 로드맵 자동 생성 | 핵심 가치 제안 |
| AI 멘토 질문·답변 | 두 번째 핵심 가치 제안 |
| 학습 맥락 자동 첨부 (LCS) | **차별화 가설(H2) 검증의 핵심** |
| 기본 Q&A 게시판 | 커뮤니티 기능의 최소 형태 |
| 결제 통합 (토스페이먼츠) | 유료 전환 가설(H1) 검증 |
| 가입·로그인 (OAuth) | 사용자 식별 필수 |

8주 알파에 **의도적으로 빼는 것**:

| 기능 | 빠지는 이유 | 후속 시점 |
|------|----------|:--------:|
| 평판·배지 시스템 | 커뮤니티 규모 작아 의미 없음 | 베타 후반 (M3+) |
| AR 공간 학습 | 기술 복잡도 높음, 핵심 가설과 무관 | Phase 2 (M12+) |
| 모바일 앱 | 웹 우선, 모바일은 반응형으로 대응 | Phase 2 (M6+) |
| 스터디 매칭 | 커뮤니티 규모 작아 의미 없음 | 베타 후반 (M3+) |
| 실습 환경 (Sandbox Runner) | 기술 복잡도, 인프라 비용 | 베타 후반 (M2+) |
| B2B 관리자 패널 | 개인 B2C 검증 우선 | M12+ |
| 자동 보정 시스템 | 데이터 축적 후 가능, 초기는 규칙 기반만 | M3+ |

**이 의도적 배제 자체가 멘토에게 보여줄 수 있는 성숙도**다. "모든 걸 다 만들려는" 신입 창업자가 아니라 "검증 가능한 최소 범위를 안다"는 신호.

### 1-3. MVP의 한계 자기 인정

8주 알파에는 다음 한계가 있다. **멘토에게 솔직하게 미리 공개**한다.

- AI 답변 품질은 Claude API의 기본 성능에 의존, 도메인 특화 fine-tuning 없음
- 사용자 100명 규모, 통계적 유의미성에 한계
- 부산 거주 창업자 단독 개발, 사용자 인터뷰는 원격 중심
- 디자인은 외주 1명에 의존, 브랜드 시스템 미완

**왜 한계를 공개하는가**: 멘토는 한계를 모르는 척하는 창업자보다 인지하고 대응하는 창업자를 신뢰한다. "성장하는 모습"의 본질은 한계 인정에서 시작된다.

## 2️⃣ 검증할 가설 3가지

### 가설 H1: 지불 의사

**가설 진술**: 개발 학습자 100명 중 5명 이상은 "맞춤 로드맵 + AI 멘토 + 학습 맥락 Q&A" 패키지에 월 14,900원을 지불할 의사가 있다.

**검증 방법**:
- 8주 알파 베타 사용자 100명 모집
- 7일 무료 체험 후 자동 유료 전환 시점에 결제 페이지 노출
- 결제 페이지 도달율, 결제 완료율 추적

**성공 기준**:
- 🟢 유료 전환 5% 이상 → 가격·BM 가설 검증, 정식 런칭 진행
- 🟡 유료 전환 3~5% → 가격 또는 가치 제안 조정 필요, 추가 인터뷰
- 🔴 유료 전환 3% 미만 → BM 근본 재설계 (가격 인하, 무료 영구화 + B2B 전환 등)

**측정 도구**: Google Analytics + 자체 Funnel 추적

### 가설 H2: LCS 효과

**가설 진술**: 학습 맥락이 자동 첨부된 AI 답변은 그렇지 않은 답변보다 사용자 만족도가 1.0점 이상 높다(5점 만점 기준).

**검증 방법**:
- 동일 사용자에게 두 종류 답변 노출 (A/B 테스트)
  - A: 학습 맥락 첨부된 AI 답변
  - B: 일반 ChatGPT 스타일 답변 (맥락 없음)
- 사용자가 두 답변을 평가 (1-5점)
- 통계적 차이 측정

**성공 기준**:
- 🟢 평균 차이 +1.0/5.0 이상 → LCS가 핵심 차별화로 입증됨
- 🟡 평균 차이 +0.5~+1.0 → 차별화는 있으나 약함, LCS 고도화 필요
- 🔴 차이 +0.5 미만 → LCS 가치 가설 재검토, 다른 차별화 모색

**측정 도구**: 자체 A/B 테스트 시스템, 답변 평가 데이터베이스

### 가설 H3: 1인 창업자의 실행력

**가설 진술**: 풀스택 역량을 가진 1인 창업자가 외주 디자인 1명 협업으로 8주 안에 동작하는 알파 MVP를 만들 수 있다.

**검증 방법**:
- 8주 스프린트 일정 준수 여부
- 매주 GitHub 커밋 및 진척 기록

**성공 기준**:
- 🟢 8주 내 알파 출시 + 베타 사용자 모집 가능 → 추가 외주 없이 진행
- 🟡 8주에 알파 출시했으나 베타 모집 지연 → 마케팅 외주 검토
- 🔴 8주 후에도 알파 미완성 → 개발 외주 추가 영입, 일정 재조정

**측정 도구**: GitHub 활동 로그, 주간 진척 보고서

### 가설들 사이의 우선순위

세 가설이 동시 검증되면 좋지만, 우선순위는:

1. **H3 (실행력)** — 이게 실패하면 H1·H2 측정 자체가 불가능
2. **H2 (LCS 효과)** — 차별화 검증, 이게 실패하면 사업 핵심 가치 의문
3. **H1 (지불 의사)** — 가격·BM 조정 가능한 영역

**즉, 8주 동안 일단 H3 달성에 집중하고, H1·H2는 알파 완성 후 베타 단계에서 측정**.

### 3-2. 핵심 사용자 흐름 5가지

**시나리오 1: 가입부터 첫 학습까지 (10분)**

```
랜딩 페이지 도착 
→ "내 실력 진단받기" 클릭
→ Google 로그인 (3초)
→ 약관 동의 (1분)
→ 진단 테스트 시작 (8분, 15문항)
→ 결과 화면: "당신은 백엔드 중급 직전, JPA가 약점"
→ AI 로드맵 자동 생성 (30초 로딩)
→ "3개월 로드맵" 제시 (총 12단계)
→ 첫 번째 콘텐츠로 진입
```

**시나리오 2: AI 멘토에게 질문 (3분)**

```
콘텐츠 학습 중 막힘
→ "AI 멘토에게 질문" 버튼
→ 채팅 화면: 학습 맥락이 자동 표시됨
   ("당신은 현재 'JPA N+1 문제 해결' 학습 중")
→ 질문 입력: "Lazy loading이 뭔가요?"
→ AI 답변 (맥락 인지): "지금 학습 중인 N+1 문제와 직접 연결됩니다..."
→ 사용자가 답변 평가 (👍 / 👎 / 더 자세히)
```

**시나리오 3: 커뮤니티 질문 작성 (5분)**

```
AI 답변으로 부족함
→ "커뮤니티에 묻기" 버튼
→ Q&A 작성 화면: 학습 맥락이 자동 첨부됨
   (현재 콘텐츠, 진도, 평판, 활성 태그)
→ 제목·본문 작성
→ 발행 → AI가 즉시 시드 답변 (대기시간 0)
→ 24시간 내 다른 답변자 답변 도착
```

**시나리오 4: 무료 → 유료 전환 (2분)**

```
7일 사용 후 AI 멘토 횟수 제한 도달
→ "프리미엄 업그레이드" 모달
→ 가격 안내 (월 14,900원)
→ "무료 7일 더" or "유료 시작" 선택
→ 유료 선택 시 토스페이먼츠 결제 화면
→ 결제 완료 → 즉시 무제한 사용
```

**시나리오 5: 마이페이지 (1분)**

```
대시보드 우상단 프로필 클릭
→ 진행 중 로드맵 상태
→ 완료한 콘텐츠 (X/Y)
→ 작성한 질문·답변 수
→ 구독 정보 + 결제 관리
→ 설정 (알림, 개인정보)
```

### 3-3. 기술 스택

| 영역 | 기술 | 선택 이유 |
|------|------|---------|
| 백엔드 | Spring Boot 4, Java 21 | 창업자 가장 자신 있는 스택 |
| 데이터베이스 | PostgreSQL 16 + pgvector | 관계형 + 벡터 검색 통합 |
| 캐시 | Redis 7 | LCS 실시간 맥락 캐시 |
| 메시징 | Apache Kafka (KRaft) | 학습 이벤트 스트림 |
| 검색 | Elasticsearch | Q&A 검색 (베타에서 추가) |
| 프론트엔드 | Flutter 3.x (Dart) + Riverpod 3 | 웹·모바일 단일 코드베이스 |
| AI | Claude API (Sonnet 4.6) | 한국어 답변 품질 + RAG |
| 결제 | 토스페이먼츠 | 국내 결제 표준 |
| 호스팅 | NCP 또는 AWS 한국 리전 | 데이터 주권 |
| 컨테이너 | Docker + Kubernetes | StockPilot 경험 재활용 |
| CI/CD | GitHub Actions + ArgoCD | 자동 배포 |

**기술 멘토에게 검증 요청할 부분**:
1. Claude API 비용 최적화 전략 (사용자당 월 사용량 캡 설정)
2. LCS 아키텍처의 보안 취약점 검토 (특히 에러 로그 sanitization)
3. PostgreSQL + pgvector vs 전용 벡터 DB(Weaviate 등) 선택 근거 검증

- "맥락이 있는지 몰랐다" 다수: 옵션 1 (UX 개선)
- "차이를 못 느꼈다" 다수: 옵션 3 (근본 품질)
- "있어도 그만, 없어도 그만": 옵션 2 (다른 가치 모색)

**H3 실패 (8주 알파 미완성)**

옵션 1: **기능 축소** → 진단·로드맵만 출시, AI 멘토는 베타 후반
옵션 2: **개발 외주 추가** → 백엔드 1명 추가 영입 (예비비 사용)
옵션 3: **일정 연기** → 알파 출시 2~3주 지연, 솔직 공개

**선택 기준**: 8주차 시점 진척 비율
- 80% 이상: 옵션 1 (기능 축소로 일정 사수)
- 60~80%: 옵션 2 (외주 영입)
- 60% 미만: 옵션 3 (현실 수용 + 멘토 협의)

### 7-3. "성공의 정의"의 유연성

8주 후 평가는 단순한 "성공/실패" 이분법이 아닙니다. 멘토는 다음 두 가지를 평가합니다:

1. **결과**: KPI 달성 여부
2. **과정**: 어려움에 어떻게 대응했는가

KPI 미달이어도 **명확한 대응**과 **다음 단계 계획**이 있으면 멘토는 긍정 평가합니다. 반대로 KPI 달성이어도 **운에 의존한 결과**면 회의적입니다.

**적극적으로 공유**:
- 매주 진척 보고서
- 가설 검증 결과 (좋든 나쁘든)
- 외부 피드백 (사용자 인터뷰)
- 의사결정의 근거

**의도적으로 페이스 조절**:
- 이전에 만든 15개 문서 전체 (한 번에 X, 필요할 때마다 점진적으로)
- 미래 비전 (AR 학습 등 너무 큰 그림)
- 너무 완성된 모습 (성장 여지 보여줘야 함)

**절대 공유 금지**:
- 다른 멘티 팀에 대한 비교나 평가
- 미공개 보안 취약점
- 다른 지원사업 결과 자랑

### 8-4. 멘토 활용을 통한 1라운드 평가 우위 확보

운영진의 평가 기준을 다시 보면 "각 멘토는 자신이 담당한 팀 중 가장 열정적이고 가능성이 높은 최우수 1인을 직접 추천하게 됩니다." 

즉 **같은 멘토 아래의 다른 팀들과 직접 경쟁**입니다. 다음 4가지로 차별화:

1. **사전 준비도**: 첫 미팅 전에 이미 풍부한 자료 보유 (단, 점진적 공개)
2. **실행 속도**: 매주 가시적 진척
3. **솔직함**: 어려움과 한계를 숨기지 않는 태도
4. **조언 반영**: 멘토 제안을 다음 미팅까지 실행

특히 4번이 핵심입니다. 멘토 입장에서 "내가 한 말이 실제로 반영된다"는 경험은 강력한 추천 동기를 만듭니다.
